# API Inquiry v0.1.1 Alpha 打磨设计

## 概要

v0.1.1 是基于现有 alpha 的聚焦打磨版本。当前应用已经可以作为 DeepSeek 余额菜单栏工具使用，DMG 也已经在多台 Mac 上验证通过。本版本重点改善首次使用体验、让错误状态具备可恢复操作、优化 API Key 管理细节，并提升免费 GitHub DMG 分发的可信度。

这不是本地 DeepSeek 控制台版本。本版本应继续保持当前轻量菜单栏形态，避免扩大产品范围。

## 目标

- 让还没有配置 DeepSeek API Key 的用户更容易理解首次启动流程。
- 刷新失败时，给用户明确的下一步操作。
- 在保持 API Key 安全边界不变的前提下，优化保存、更换和删除反馈。
- 通过 UI 体现刷新中状态，让手动刷新更有响应感。
- 统一 v0.1.1 在 app 元数据、DMG 打包、release notes 和文档中的版本口径。
- 为 DMG 生成 SHA-256 checksum，并记录用户如何校验。
- 开发过程态保持快速迭代：使用本地启动脚本验证，完整 DMG 打包只放在 release 验证阶段。

## 非目标

- v0.1.1 不构建本地 DeepSeek 控制台。
- 不增加详细用量图表。
- 不增加新的 API 供应商。
- 不引入 Developer ID 签名或 Apple notarization 公证。
- 不整体重设计面板，也不替换当前极简视觉语言。
- 暂不做完整应用本地化。文档继续保持中英双语，UI 文案本版本可以继续使用简洁英文。
- 不为每个开发中间状态打包 DMG。

## 当前基线

- 菜单栏标签显示 DeepSeek 图标和紧凑余额。
- 详情面板显示 DeepSeek logo、大号余额、状态、最近更新时间、API Key 控件、AutoStart、Console 和 Quit。
- API Key 存储在 macOS Keychain，保存后不再明文展示。
- 应用每 5 分钟自动刷新，并支持手动刷新。
- Release DMG 包含 `API Inquiry.app` 和 `Applications` 快捷方式，app 在 Finder 中可见。
- Alpha 分发路径是 GitHub Releases，使用 ad-hoc 签名，不做 Apple 公证。

## 产品设计

### 首次启动设置

当没有配置 API Key 时，面板应明确告诉用户下一步要做什么，但不能变成冗长的 onboarding 页面。

推荐行为：

- 菜单栏继续保持紧凑 setup 状态。
- 在面板中靠近 API Key 输入框的位置显示一段简短设置说明。
- 提供直接的 `Open DeepSeek Console` 或 `Get API Key` 操作。
- 保持 secure text field 和主要 `Save` 操作可见。
- 保存成功后清空输入、折叠 API Key 区域，并刷新余额。
- 显示短暂成功反馈，例如 `Saved securely`。

设置说明要足够短，保持菜单栏工具的轻量感。

### 可操作错误状态

错误信息不仅要说明发生了什么，还要告诉用户下一步可以做什么。

推荐行为：

- 认证失败：显示清晰的 API Key 警告，并提供 `Replace Key` 和 `Delete Key`。
- 缺少 key：显示设置状态，而不是泛化失败。
- 网络失败：如果已有上一次成功余额，则继续保留，并提供 `Retry`。
- 限流或服务器错误：显示简短的重试导向提示，并提供 `Retry`。
- 解码失败或异常响应：显示刷新失败，并提供 `Retry`。

为了避免脆弱的字符串判断，核心状态应暴露错误类别或失败原因，让 ViewModel 基于类别决定可用操作。

### API Key 管理

现有安全策略保持不变：应用永远不明文展示已保存的 API Key。

推荐行为：

- 已配置状态默认保持折叠。
- 展开 API Key 行后显示更换用的 secure field、`Replace` 和 `Delete`。
- `Delete` 删除前需要用户确认。
- Replace 失败时保留用户输入，方便修正。
- Replace 成功后清空输入、折叠 API Key 行，并显示短暂成功反馈。
- 设置相关消息不得包含 API Key 或 Authorization 头。

### 刷新反馈

刷新按钮应反映当前加载状态。

推荐行为：

- 刷新进行中时禁用刷新按钮。
- 状态行继续显示 `Refreshing`。
- 避免在菜单栏或面板里加入过于吵闹的动画。
- 保留当前 controller 层防止重叠刷新的保护。

### 面板结构

当前面板继续保持紧凑。v0.1.1 可以做轻量重排或条件展开，但不应变成完整设置窗口。

推荐行为：

- 正常 loaded 状态下，API Key 管理保持次要层级。
- setup 或认证失败状态下，让 API Key 控件更突出。
- 底部继续保持三个紧凑模块：`AutoStart`、`Console` 和 `Quit`。
- 更深入的 AutoStart 设置和完整 Settings UI 延后处理，除非错误恢复能力确实需要。

## 发布与打包设计

### 版本一致性

v0.1.1 应使用单一版本源来驱动 release 打包。

推荐行为：

- app 的 `CFBundleShortVersionString` 设置为 `0.1.1`。
- app 的 `CFBundleVersion` 设置为匹配的 build number，初始为 `2`。
- release tag 使用类似 `v0.1.1-alpha.1`。
- DMG 文件名使用版本化形式，例如 `API-Inquiry-v0.1.1-alpha.1.dmg`。
- 只有在用户明确希望 GitHub 下载文件名保持稳定时，才继续保留 `API-Inquiry-alpha.dmg`。如果两者都生成，需要在文档中说明用户应该下载哪个。

### Checksum

Release 流程应在 DMG 旁边生成 SHA-256 checksum 文件。

推荐输出：

```text
dist/API-Inquiry-v0.1.1-alpha.1.dmg
dist/API-Inquiry-v0.1.1-alpha.1.dmg.sha256
```

README 和 release notes 应包含类似命令：

```bash
shasum -a 256 -c API-Inquiry-v0.1.1-alpha.1.dmg.sha256
```

### DMG 校验

打包脚本应继续校验最终 DMG，而不仅是校验 staging 目录。

校验应覆盖：

- 挂载后的 DMG 中存在 `API Inquiry.app`。
- `API Inquiry.app` 没有 hidden 标记。
- 存在 `Applications` 快捷方式。
- app 可执行文件存在。
- app Info.plist 中 bundle id 和版本符合预期。
- ad-hoc code signature 校验通过。
- checksum 文件能校验生成的 DMG。

### 开发验证与发布验证

开发过程态优先使用快速本地启动流程。

推荐行为：

- 使用 `Scripts/build-local-app.sh` 和本地 `open` 命令进行 UI 测试。
- 不为每个中间改动生成 DMG。
- 使用 `swift run APIInquiryCoreTestsRunner` 验证核心行为。
- 只有准备 v0.1.1 release candidate 时，才执行完整 app 打包、DMG 生成、checksum 生成和 release notes 检查。

## 架构说明

### 错误分类

当前失败状态主要暴露用户可见文本。v0.1.1 应增加一个小型内部分类层，让 UI 不需要解析字符串就能决定恢复动作。

可能的模型：

- `notConfigured`
- `authenticationFailed`
- `rateLimited`
- `networkUnavailable`
- `serverError`
- `decodingFailed`
- `unknown`

展示文案可以继续保持友好简短，而类别用于驱动 `Retry`、`Replace Key` 或 `Delete Key` 等操作。

### ViewModel 职责

`MenuBarBalanceViewModel` 继续负责 UI-facing 状态。它可以暴露少量派生属性：

- 当前是否应禁用刷新
- 是否显示 setup 引导
- 当前可用的恢复动作
- API Key 保存/删除反馈
- 后续需要时可暴露版本展示或诊断元数据

View 应继续保持声明式，避免重复业务规则。

### 打包脚本

打包脚本应避免硬编码版本漂移。如果能保持简单，可以引入一个小型共享版本文件或共享脚本变量。

Release 工作流当前继续以本地优先为准。如果后续手工发布步骤变得容易出错，再考虑加入 GitHub Actions 自动化。

## 测试计划

### 核心测试

- 缺少 API Key 时进入 setup 状态。
- 认证失败时暴露更换/删除 key 的恢复动作。
- 网络、限流、服务器和解码失败时暴露 retry 操作。
- 刷新中状态禁用手动刷新。
- API Key 保存成功后清空输入并产生成功反馈。
- API Key 保存失败时保留输入并显示安全错误。
- 删除 API Key 通过 UI 确认路径后执行，并在确认后将应用标记为未配置。
- 所有测试不得暴露或记录真实 API Key。

### 打包测试

- 构建本地 app，用于快速开发验证。
- 运行 `swift run APIInquiryCoreTestsRunner`。
- 仅 release candidate 阶段运行完整 DMG 打包脚本。
- 挂载生成的 DMG 并验证 app 可见。
- 验证 Info.plist 版本和 bundle id。
- 验证 ad-hoc code signature。
- 验证 SHA-256 checksum 文件。

### 手动测试

- 在没有 API Key 的状态下首次启动。
- 本地保存真实 API Key。
- 确认余额刷新和菜单栏显示。
- 触发手动刷新并确认按钮状态。
- 测试无效 API Key，并确认恢复动作可用。
- 取消删除 API Key，确认 key 仍然保持配置。
- 确认删除 API Key 后回到 setup 状态。
- 从底部打开 Console。
- 确认 AutoStart 切换仍然可用。
- 开发过程中通过本地脚本启动 in-progress app。
- 只在 release candidate 验证阶段从 DMG 安装。

## 推进计划

1. 编写并审阅本设计文档的英文版和中文版。
2. 设计文档确认后，创建 v0.1.1 实施计划。
3. 先实现 UI 和状态模型打磨。
4. 再实现打包、版本和 checksum 变更。
5. 开发验证使用本地 app 启动。
6. Release candidate 验证阶段才执行完整 DMG 打包。
7. 准备 v0.1.1 中英文 release notes。
8. 本地验证和 DMG 验证通过后，发布 v0.1.1 alpha。
