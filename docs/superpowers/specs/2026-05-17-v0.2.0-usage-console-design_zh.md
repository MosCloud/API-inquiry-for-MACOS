# API Inquiry v0.2.0 用量控制台设计

## 概要

v0.2.0 将 API Inquiry 从单一 DeepSeek 余额菜单栏工具扩展为“余额 + 本地用量控制台”。现有菜单栏继续负责快速查看余额、刷新状态和打开控制台；新的本地控制台负责 API Key 管理、DeepSeek 用量官方导出包导入、本地用量汇总和明细查看。

本版本采用官方导出文件优先路线。DeepSeek 公开文档中清晰可用的详细用量路径是从 Usage 页面导出用量文件；实际下载结果是一个 zip，内含 `cost-*.csv` 和 `amount-*.csv`。v0.2.0 不尝试网页抓取，也不假设存在稳定的详细用量 API。用户从 DeepSeek Usage 页面下载官方 zip 后，在本地控制台中一次导入并分析。

## 目标

- 新增本地控制台窗口，作为配置和用量分析的集中入口。
- 将 API Key 管理从菜单栏详情面板移动到控制台中。
- 支持导入 DeepSeek Usage 官方 zip 导出包，并在本机解析、归一化和保存用量记录；保留兼容 CSV 兜底。
- 展示用量总览，包括总费用、请求数、输入 tokens、输出 tokens 和总 tokens。
- 展示按模型汇总的用量，以及可扫描的明细记录列表。
- 让菜单栏详情面板保持轻量，只负责状态展示、刷新、打开控制台、AutoStart 和退出。
- 维持现有安全边界：API Key 只存储在 macOS Keychain，用量导出数据只保存在本机。
- 为后续图表、月份历史、多供应商和自动化导入打好数据模型基础。

## 非目标

- v0.2.0 不抓取或自动化 DeepSeek 网页控制台。
- 不新增非 DeepSeek 供应商。
- 不实现复杂图表库、趋势图或交互式可视化；本版本只做文本和表格式汇总。
- 不把原始导出文件复制进应用数据目录；只保存归一化后的本地记录和导入元数据。
- 不做云同步、账号体系或跨设备共享。
- 不引入 Developer ID 签名或 Apple notarization 公证。
- 不做完整应用本地化。UI 文案可以继续使用简洁英文，项目文档保持中英双语。

## 当前基线

- `v0.1.1` 已发布，菜单栏可以显示 DeepSeek 余额并每 5 分钟刷新。
- API Key 存储在 Keychain 中，保存后不会明文展示。
- 菜单栏详情面板中包含 API Key 设置、更换、删除、AutoStart、Console 和 Quit。
- 当前 `Console` 操作直接打开 DeepSeek Usage 网页。
- 打包脚本已经支持版本化 DMG 和 SHA-256 checksum。
- 测试使用本地 `APIInquiryCoreTestsRunner`，不依赖 XCTest。

## 产品设计

### 菜单栏详情面板

菜单栏详情面板在 v0.2.0 中回到轻量状态面板，不再承载完整 API Key 表单。

推荐行为：

- 保留 DeepSeek logo、余额、状态、最近刷新时间和手动刷新按钮。
- 保留底部 `AutoStart`、`Console` 和 `Quit` 三个紧凑操作模块。
- `Console` 打开本地 API Inquiry 控制台窗口，而不是直接打开 DeepSeek Usage 网页。
- 没有配置 API Key 时，面板显示简短 setup 状态，并提供 `Open Console` 操作。
- API Key 认证失败时，面板显示错误摘要和 `Open Console` 操作；更换和删除 key 的具体操作在控制台中完成。
- 网络失败、限流和服务器错误仍可在面板中提供 `Retry`，因为这些恢复动作不需要进入设置。

这让菜单栏成为“看状态”和“进入控制台”的地方，避免把设置、导入和分析都塞进 320px 宽的弹出面板。

### 本地控制台窗口

v0.2.0 新增一个原生 macOS 控制台窗口。控制台是 API Inquiry 的集中管理界面，包含三个主要区域：

- `Overview`：显示余额状态、最近刷新时间、用量总览和最近导入信息。
- `Usage`：导入 DeepSeek 官方用量导出包、查看模型汇总和明细列表。
- `API`：配置、替换和删除 DeepSeek API Key，并提供打开 DeepSeek 官方 Usage 页面的外部链接。

控制台可以使用 SwiftUI `WindowGroup` 或轻量 `NSWindowController` 实现。具体实现应保持单窗口复用：用户多次点击菜单栏 `Console` 时，前置已有控制台窗口，而不是不断创建重复窗口。

### API Key 管理迁移

API Key 管理从 `MenuBarContentView` 移动到控制台中。

推荐行为：

- 控制台 `API` 区显示当前 key 状态：`Configured` 或 `Not configured`。
- 未配置状态下，显示 secure text field、`Save` 和 `Open DeepSeek Console`。
- 已配置状态下，默认不显示已保存 key，只显示 `Replace Key` 和 `Delete Key`。
- `Replace Key` 展开 secure text field；失败时保留输入，成功时清空输入并刷新余额。
- `Delete Key` 需要内联确认；确认后从 Keychain 删除，并让余额状态回到 setup。
- 所有反馈文案不得包含 API Key、Authorization header 或原始错误中可能泄露凭据的内容。

为了减少重复业务规则，API Key 保存、替换和删除仍应通过 Core 中现有的 `CredentialStore` 和刷新控制器路径执行。菜单栏和控制台共享同一个 app-level view model 或共享同一个 controller 实例。

### 官方导出导入

用户在控制台 `Usage` 区选择 DeepSeek Usage 下载的官方 zip 导出包。应用读取文件一次，不保留对原文件的长期访问。为兼容早期假数据和通用测试，控制台仍可导入符合字段要求的单个 CSV。

v0.2.0 的导入策略：

- 每次成功导入会替换当前本地用量数据集。
- 这避免早期版本出现重复导入、跨月合并和记录去重的隐性复杂度。
- 后续版本可以在当前归一化模型基础上增加按月份追加和合并历史。

官方 zip 解析规则：

- 识别 `cost-*.csv`：读取日期、模型、费用和货币代码。
- 识别 `amount-*.csv`：读取日期、模型、指标类型和数量。
- 按 `UTC 日期 + 模型` 合并两张表。
- 将 `request_count` 映射为请求数。
- 将 `input_cache_hit_tokens` 和 `input_cache_miss_tokens` 合并为输入 tokens。
- 将 `output_tokens` 映射为输出 tokens。
- 用输入 tokens + 输出 tokens 计算总 tokens。
- 不保存 `user_id`、`api_key_name` 或遮罩后的 `api_key`。

兼容 CSV 解析仍采用 header-driven 方式，不依赖固定列顺序。解析器接受一组明确的列别名，并把数据归一化为本地 `UsageRecord`。兼容 CSV 至少支持这些语义字段：

- 日期或时间
- 模型
- 请求数
- 输入 tokens
- 输出 tokens
- 总 tokens
- 费用金额
- 货币代码

如果官方 zip 缺少 `cost` 或 `amount` CSV，导入失败并显示可操作错误。兼容 CSV 缺少日期、模型和费用这类必要字段时，导入失败并显示可操作错误，例如 `The CSV is missing required usage columns.`。未知列可以忽略，但缺少必要列、无法解析数字、无法解析日期时必须给出明确错误。测试 fixture 必须使用假数据，不包含真实账号、真实 API Key 或真实账单。

### 用量总览与明细

导入成功后，控制台应展示：

- 当前数据集的时间范围。
- 总费用，优先使用 CSV 中的货币代码；多货币时显示 `Mixed`，不做自动汇率换算。
- 请求数合计。
- 输入 tokens、输出 tokens 和总 tokens 合计。
- 按模型聚合的费用、请求数和 tokens。
- 明细列表，包含日期、模型、费用、请求数和 tokens。

明细列表应适合扫描，不追求复杂筛选。v0.2.0 可以支持按日期倒序展示和基础空状态；搜索、排序、多月份筛选和图表放到后续版本。

### 本地存储

用量数据保存到用户本机的 Application Support 目录，例如：

```text
~/Library/Application Support/API Inquiry/usage-records.json
```

存储内容只包括归一化后的用量记录、导入时间、源文件名和解析版本。不保存 API Key，不保存 Authorization header，不复制原始导出文件。

控制台提供 `Clear Usage Data` 操作，删除本地用量记录但不影响 Keychain 中的 API Key。

## 架构设计

### Core 模型

在 `APIInquiryCore` 中增加用量相关模型：

- `UsageRecord`：单条归一化用量记录。
- `UsageDataset`：当前本地数据集，包含 records 和 import metadata。
- `UsageImportSummary`：导入结果摘要。
- `UsageTotals`：总费用、请求数和 tokens 合计。
- `UsageModelSummary`：按模型聚合结果。
- `UsageImportError`：缺失列、日期解析失败、数字解析失败和空文件等错误。

这些模型不依赖 SwiftUI，方便在本地 runner 中测试。

### 用量导入器

新增 `DeepSeekUsageFileImporter`、`DeepSeekUsageExportParser` 和 `DeepSeekUsageCSVParser`，负责：

- 读取官方 zip 中的 `cost-*.csv` 和 `amount-*.csv`。
- 根据 header 映射字段。
- 将官方双表或兼容单 CSV 转换为 `UsageRecord`。
- 跳过完全空白行。
- 对必要字段缺失或格式错误抛出 `UsageImportError`。

如果 Swift 标准库没有合适 CSV 解析能力，项目可以实现一个小型 RFC 4180 风格解析器，至少覆盖逗号、换行和双引号转义。不要用简单 `split(separator: ",")` 处理 CSV。

### 用量存储

新增 `UsageDataStore` 协议和本地 JSON 实现：

- `loadDataset()`
- `saveDataset(_:)`
- `clearDataset()`

测试使用内存实现，避免写入真实 Application Support。

### 控制台 ViewModel

新增控制台用 view model，负责组合：

- 当前余额状态。
- API Key 配置状态和操作。
- 用量导入状态、错误和成功反馈。
- 用量总览、模型汇总和明细列表。

菜单栏 view model 应从“API Key 表单控制者”收缩为“余额状态面板控制者”。API Key 的业务操作移动到控制台 view model，但底层仍复用同一个 credential store 和 refresh controller。

### App 层

App 层需要支持打开本地控制台窗口：

- 菜单栏 `Console` 操作打开本地控制台。
- 控制台内提供 `Open DeepSeek Usage` 外部链接，打开 `https://platform.deepseek.com/usage`。
- 如果用户在 setup 或认证失败状态点击菜单栏入口，控制台应默认显示 API 区。
- 如果用户只是查看用量，控制台默认显示 Overview 或 Usage 区。

## 错误处理

- 用量文件无法读取：显示文件读取失败，并建议重新选择文件。
- 官方 zip 缺少必要 CSV：显示缺少必要导出文件。
- CSV 缺少必要列：显示缺少必要用量列。
- 日期无法解析：指出日期列无法解析，不回显整行原始数据。
- 数字无法解析：指出金额或 tokens 列无法解析，不回显敏感上下文。
- 空 CSV：显示文件中没有可导入的用量记录。
- 存储失败：显示本地保存失败，并保留内存中的解析结果直到用户关闭控制台或重新导入。
- API Key 保存失败：保留输入并显示安全错误。
- API Key 删除失败：显示安全错误，并保持 UI 状态不误报删除成功。

## 测试计划

核心测试：

- 官方 zip 导入器正确合并 `cost-*.csv` 和 `amount-*.csv`。
- CSV parser 正确解析带引号、逗号和换行的字段。
- CSV parser 不依赖列顺序。
- 缺少必要列时导入失败。
- 日期、金额和 tokens 解析失败时导入失败。
- 空 CSV 导入失败。
- 汇总逻辑正确计算费用、请求数、输入 tokens、输出 tokens 和总 tokens。
- 多模型聚合结果正确。
- 本地 store 可以保存、读取和清空数据集。
- 控制台 view model 可以触发导入、产生摘要、清空数据和暴露安全错误。
- 菜单栏 view model 不再暴露完整 API Key 编辑状态给菜单栏面板。

手动测试：

- 从菜单栏打开本地控制台窗口。
- 无 key 状态下从菜单栏进入控制台并保存 key。
- 已配置 key 状态下在控制台替换和删除 key。
- 从控制台打开 DeepSeek 官方 Usage 页面。
- 导入假官方 zip 后查看总览、模型汇总和明细。
- 重新导入另一份用量文件后确认本地数据被替换。
- 清空本地用量数据后确认 API Key 和余额刷新不受影响。
- 打包 DMG 并校验 checksum。
