# API Inquiry v0.2.0 Usage Console Design

## Summary

v0.2.0 expands API Inquiry from a DeepSeek balance menu bar utility into a balance plus local usage console app. The menu bar remains responsible for quick balance visibility, refresh status, and opening the console. The new local console owns API key management, DeepSeek usage CSV import, local usage summaries, and usage detail inspection.

This release takes a CSV-first path. The clearly available detailed usage workflow in DeepSeek's public documentation is exporting CSV from the Usage page, so v0.2.0 does not scrape the website and does not assume a stable detailed usage API exists. Users export a CSV from DeepSeek Usage, then import and analyze it locally.

## Goals

- Add a local console window as the centralized place for configuration and usage analysis.
- Move API key management out of the menu bar details panel and into the console.
- Support importing DeepSeek Usage CSV files, parsing them locally, normalizing records, and saving them on the user's Mac.
- Show usage totals, including cost, request count, input tokens, output tokens, and total tokens.
- Show model-level usage summaries and a scannable detail table.
- Keep the menu bar details panel lightweight: status, refresh, open console, AutoStart, and quit.
- Preserve the current safety boundary: API keys only live in macOS Keychain, and CSV-derived usage data only lives locally.
- Establish data model foundations for future charts, month history, multiple providers, and automated imports.

## Non-Goals

- Do not scrape or automate the DeepSeek web console in v0.2.0.
- Do not add non-DeepSeek providers.
- Do not add complex charting libraries, trend charts, or interactive visualization; this release uses textual and tabular summaries only.
- Do not copy the original CSV file into the app data directory; only normalized local records and import metadata are saved.
- Do not add cloud sync, accounts, or cross-device sharing.
- Do not introduce Developer ID signing or Apple notarization.
- Do not add full app localization yet. UI copy can stay concise English, while project documentation remains bilingual.

## Current Baseline

- `v0.1.1` is released, and the menu bar can display the DeepSeek balance and refresh every 5 minutes.
- API keys are stored in Keychain and are not shown after saving.
- The menu bar details panel includes API key setup, replace, delete, AutoStart, Console, and Quit.
- The current `Console` action opens DeepSeek's Usage web page directly.
- Packaging scripts already support versioned DMGs and SHA-256 checksums.
- Tests use the local `APIInquiryCoreTestsRunner` and do not depend on XCTest.

## Product Design

### Menu Bar Details Panel

In v0.2.0, the menu bar details panel returns to being a lightweight status panel and no longer carries the full API key form.

Recommended behavior:

- Keep the DeepSeek logo, balance, status, last refresh time, and manual refresh button.
- Keep the footer actions as three compact modules: `AutoStart`, `Console`, and `Quit`.
- `Console` opens the local API Inquiry console window instead of directly opening the DeepSeek Usage web page.
- When no API key is configured, the panel shows a short setup state and an `Open Console` action.
- When API key authentication fails, the panel shows an error summary and an `Open Console` action. The concrete replace and delete key actions happen in the console.
- Network failures, rate limits, and server errors can still expose `Retry` in the panel because those recovery actions do not require settings.

This makes the menu bar the place to see state and enter the console, instead of packing settings, import, and analysis into a 320px-wide popover.

### Local Console Window

v0.2.0 adds a native macOS console window. The console is the central management surface for API Inquiry and contains three main areas:

- `Overview`: balance state, last refresh time, usage totals, and latest import information.
- `Usage`: import DeepSeek CSV files, inspect model summaries, and review detail records.
- `API`: configure, replace, and delete the DeepSeek API key, plus an external link to the official DeepSeek Usage page.

The console can be implemented with SwiftUI `WindowGroup` or a lightweight `NSWindowController`. The implementation should reuse a single window: clicking `Console` repeatedly should bring the existing console forward instead of creating duplicate windows.

### API Key Management Migration

API key management moves from `MenuBarContentView` into the console.

Recommended behavior:

- The console `API` area shows the current key status: `Configured` or `Not configured`.
- In the not configured state, show a secure text field, `Save`, and `Open DeepSeek Console`.
- In the configured state, never show the saved key; show `Replace Key` and `Delete Key`.
- `Replace Key` expands a secure text field. Failure keeps the typed input; success clears the input and refreshes balance.
- `Delete Key` requires inline confirmation. After confirmation, remove the key from Keychain and return balance state to setup.
- Feedback copy must never include the API key, authorization header, or raw errors that could leak credentials.

To reduce duplicated business rules, API key save, replace, and delete should still go through the existing `CredentialStore` and refresh controller paths in Core. The menu bar and console share one app-level view model or one shared controller instance.

### CSV Import

Users choose a DeepSeek Usage CSV file from the console `Usage` area. The app reads the file once and does not keep long-term access to the original file.

v0.2.0 import strategy:

- Every successful import replaces the current local usage dataset.
- This avoids early hidden complexity around duplicate imports, cross-month merging, and record de-duplication.
- A later release can add month-based append and history merging on top of the normalized model.

CSV parsing should be header-driven and not depend on fixed column order. The parser accepts an explicit set of column aliases and normalizes rows into local `UsageRecord` values. The first version supports at least these semantic fields:

- Date or time
- Model
- Request count
- Input tokens
- Output tokens
- Total tokens
- Cost amount
- Currency code

If the CSV is missing required fields such as date, model, or cost, import fails with an actionable error such as `The CSV is missing required usage columns.` Unknown columns can be ignored, but missing required columns, invalid numbers, and invalid dates must produce clear errors. Test fixtures must use fake data and must not contain real accounts, real API keys, or real billing data.

### Usage Totals And Details

After a successful import, the console should show:

- The imported dataset's date range.
- Total cost, using the CSV currency code where possible. If multiple currencies exist, show `Mixed` and do not perform currency conversion.
- Total request count.
- Total input tokens, output tokens, and total tokens.
- Cost, request, and token totals grouped by model.
- A detail list with date, model, cost, requests, and tokens.

The detail list should optimize for scanning, not advanced filtering. v0.2.0 can support reverse chronological display and empty states. Search, sorting, multi-month filtering, and charts are deferred.

### Local Storage

Usage data is saved in the user's local Application Support directory, for example:

```text
~/Library/Application Support/API Inquiry/usage-records.json
```

Stored content only includes normalized usage records, import time, source file name, and parser version. It does not save API keys, authorization headers, or a copy of the original CSV file.

The console exposes `Clear Usage Data`, which removes local usage records without affecting the API key stored in Keychain.

## Architecture Design

### Core Models

Add usage models to `APIInquiryCore`:

- `UsageRecord`: one normalized usage record.
- `UsageDataset`: the current local dataset, including records and import metadata.
- `UsageImportSummary`: import result summary.
- `UsageTotals`: aggregate cost, requests, and token totals.
- `UsageModelSummary`: per-model aggregate result.
- `UsageImportError`: missing columns, date parse failure, number parse failure, empty file, and similar errors.

These models should not depend on SwiftUI, so they can be tested by the local runner.

### CSV Parser

Add `DeepSeekUsageCSVParser`, responsible for:

- Parsing UTF-8 CSV text.
- Mapping fields by header.
- Converting each row into `UsageRecord`.
- Skipping completely blank rows.
- Throwing `UsageImportError` for missing required fields or invalid values.

If the Swift standard library does not provide a suitable CSV parser, the project can implement a small RFC 4180-style parser covering commas, newlines, and escaped double quotes. Do not parse CSV with a simple `split(separator: ",")`.

### Usage Store

Add a `UsageDataStore` protocol and a local JSON implementation:

- `loadDataset()`
- `saveDataset(_:)`
- `clearDataset()`

Tests should use an in-memory implementation to avoid writing to the real Application Support directory.

### Console ViewModel

Add a console view model that combines:

- Current balance state.
- API key configuration state and actions.
- CSV import state, errors, and success feedback.
- Usage totals, model summaries, and detail records.

The menu bar view model should shrink from an API key form controller into a balance status panel controller. API key business actions move to the console view model, while still reusing the same credential store and refresh controller underneath.

### App Layer

The app layer needs to open a local console window:

- The menu bar `Console` action opens the local console.
- The console contains `Open DeepSeek Usage`, which opens `https://platform.deepseek.com/usage`.
- If the user opens the console from a setup or authentication failure state, the console should default to the API area.
- If the user opens the console to inspect usage, the console should default to Overview or Usage.

## Error Handling

- File read failure: show that the file could not be read and ask the user to choose it again.
- Missing required CSV columns: show that required usage columns are missing.
- Date parse failure: identify the date column as unparseable without echoing the full raw row.
- Number parse failure: identify the amount or token column as unparseable without echoing sensitive context.
- Empty CSV: show that the file has no importable usage records.
- Storage failure: show local save failure and keep the parsed result in memory until the user closes the console or imports again.
- API key save failure: keep the input and show a safe error.
- API key delete failure: show a safe error and do not incorrectly report success.

## Test Plan

Core tests:

- CSV parser correctly parses fields with quotes, commas, and newlines.
- CSV parser does not depend on column order.
- Import fails when required columns are missing.
- Import fails when date, amount, or token values cannot be parsed.
- Empty CSV import fails.
- Summary logic correctly totals cost, requests, input tokens, output tokens, and total tokens.
- Multi-model aggregation is correct.
- Local store can save, load, and clear a dataset.
- Console view model can import, expose summaries, clear data, and expose safe errors.
- Menu bar view model no longer exposes full API key editing state to the menu bar panel.

Manual tests:

- Open the local console from the menu bar.
- From no-key state, open the console and save a key.
- From configured state, replace and delete the key in the console.
- Open the official DeepSeek Usage page from the console.
- Import a fake CSV and inspect totals, model summaries, and details.
- Import another CSV and confirm local usage data is replaced.
- Clear local usage data and confirm the API key and balance refresh are unaffected.
- Package the DMG and verify the checksum.

## Release Design

- Upgrade app version to `0.2.0`.
- Increase build number from `v0.1.1`'s `2` to `3`.
- Use `API-Inquiry-v0.2.0.dmg` as the DMG filename.
- Use `API-Inquiry-v0.2.0.dmg.sha256` as the checksum filename.
- Add English and Chinese release notes that only mention changes since `v0.1.1`: usage console, CSV import, API key management migration, and local usage storage.

## Future Extensions

- Month-based append and merging for multiple CSV import histories.
- Monthly cost, request, and token trend charts.
- Search, filtering, sorting, and exporting local summaries.
- More DeepSeek CSV format variants.
- Additional API providers based on the provider abstraction.
- Developer ID signing, Apple notarization, and a more formal update flow.
