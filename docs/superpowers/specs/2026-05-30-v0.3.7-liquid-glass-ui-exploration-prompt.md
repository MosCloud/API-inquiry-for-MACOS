# v0.3.7 Liquid Glass UI Exploration Prompt

This prompt is for exploring a Liquid Glass / macOS material direction for API Inquiry in a new session. It is not an implementation plan or permission for a redesign. The goal is to test whether glass/material treatment can serve the content, not make the UI more decorative.

## Prompt For The New Session

```text
We are working in /Users/zbw/Desktop/API-inquiry/.worktrees/v0.3.7 to explore a Liquid Glass / macOS material UI direction for API Inquiry v0.3.7.

Please read first:
- AGENT.md
- task_plan.md
- findings.md
- progress.md
- docs/superpowers/plans/2026-05-30-v0.3.7-ui-polish_zh.md
- docs/superpowers/specs/2026-05-30-v0.3.7-liquid-glass-ui-exploration-prompt.md

Core principles:
- This is visual exploration, not a formal redesign.
- API Inquiry must stay minimal, restrained, efficient, and lightly technical.
- UI must serve the content. Remove or reject any glass, translucency, blur, reflection, highlight, animation, or material treatment that makes status, quota, provider, or credential information harder to scan.
- The menu bar panel is a quick status surface, not a dashboard.
- Console is a focused management surface, not an operations backend.
- Liquid Glass / material should stay behind the content layer and must not compete with foreground information.

Exploration goals:
1. Decide whether macOS material / Liquid Glass fits API Inquiry.
2. Find the smallest viable glass/material surface, not a full glass redesign.
3. Preserve current information architecture, core layouts, provider workflows, and provider architecture.
4. Produce a small, reviewable, reversible proposal or experiment.

Preferred exploration scope:
- Lightweight material on the Console window background or root container.
- Lightweight material or native hierarchy treatment for the Console top navigation.
- Evaluate whether popovers, confirmations, and menu-like surfaces should use system material.
- Provider cards, quota numbers, status badges, and API key controls should generally keep stable backgrounds and strong contrast.
- Treat the menu bar panel cautiously; do not start by redesigning it.

Hard no:
- No large glass provider cards.
- No text or quota numbers floating over complex transparent backgrounds.
- No strong refraction, strong glow, obvious edge highlights, decorative shadows, or concept-art styling.
- Do not reduce information density, contrast, or readability for the sake of glass.
- Do not turn Console into a visual dashboard.
- Do not change provider architecture, credential storage, refresh logic, or v0.4.0 provider expansion behavior.
- Do not raise the minimum supported macOS version just for new UI APIs. v0.3.7 should remain macOS 13+ by default. Availability-gated enhancements for macOS 14/15+ are allowed only with macOS 13 fallback.

Suggested workflow:
1. Read the current UI files and understand MenuBar, Console Home, Console API, and Settings hierarchy.
2. Present 2-3 Liquid Glass exploration approaches with tradeoffs.
3. Recommend the most restrained approach first: Console outer/background and navigation material only; leave Provider cards unchanged initially.
4. If implementing, keep it as a small patch or commit that can be reverted easily.
5. Run swift build after implementation; run swift run APIInquiryCoreTestsRunner if Core display state or test-covered behavior changes.
6. Launch the app for review, especially in dark/light mode, complex desktop wallpaper, Increase Contrast, and Reduce Transparency.

Acceptance criteria:
- It still immediately feels like API Inquiry, not a different app.
- Content is more prominent than the effect.
- Status and quota values are easier to read, never harder.
- Glass is background hierarchy, not the main character.
- Every visual change can be justified as improving hierarchy, scanability, or feedback, not as looking cooler.
- If review shows readability got worse, revert the glass effect before trying to beautify it further.
```

## Design Understanding Summary

- The right use of Liquid Glass for API Inquiry is lightweight environmental material, not a visual theme.
- The best candidates are window backgrounds, navigation surfaces, and transient surfaces such as popovers.
- The worst candidates are core data, provider card bodies, status badges, API key inputs, and confirmation areas.
- A successful exploration should not make the glass effect obvious; it should make the UI feel better layered while keeping content calm, clear, and scannable.
