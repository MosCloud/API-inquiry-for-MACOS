# API Inquiry v0.1.1 Alpha Polish Design

## Summary

v0.1.1 is a focused polish release for the existing alpha. The app already works as a DeepSeek balance menu bar utility and the DMG has been validated on more than one Mac. This release improves the first-run experience, makes error states actionable, tightens API key management, and makes the free GitHub DMG distribution more trustworthy.

This is not the local DeepSeek console release. It should keep the current lightweight menu bar shape and avoid broad product expansion.

## Goals

- Make first launch clearer for users who have not configured a DeepSeek API key.
- Give users an obvious next action when refresh fails.
- Keep API key handling safe while improving save, replace, and delete feedback.
- Make manual refresh feel responsive by reflecting the refreshing state in the UI.
- Unify the v0.1.1 version across app metadata, DMG packaging, release notes, and documentation.
- Generate a SHA-256 checksum for the DMG and document how users can verify it.
- Keep development iteration fast by using local launch scripts during in-progress work and reserving full DMG packaging for release validation.

## Non-Goals

- Do not build the local DeepSeek console in v0.1.1.
- Do not add detailed usage charts.
- Do not add additional API providers.
- Do not introduce Developer ID signing or Apple notarization.
- Do not redesign the whole panel or replace the current minimal visual language.
- Do not add full app localization yet. Documentation remains bilingual, while UI copy can stay concise English for this release.
- Do not package a DMG for every intermediate development change.

## Current Baseline

- The menu bar label shows the DeepSeek icon plus a compact balance value.
- The details panel shows the DeepSeek logo, large balance, status, last update time, API key controls, AutoStart, Console, and Quit.
- API keys are stored in macOS Keychain and are not shown after saving.
- The app refreshes automatically every 5 minutes and supports manual refresh.
- The release DMG includes `API Inquiry.app` and an `Applications` symlink, and the app is visible in Finder.
- The alpha distribution path is GitHub Releases with ad-hoc signing and no Apple notarization.

## Product Design

### First-Run Setup

When no API key is configured, the panel should make the next step clear without turning into a large onboarding page.

Recommended behavior:

- Keep the menu bar item in the compact setup state.
- In the panel, show a short setup message near the API key field.
- Offer a direct `Open DeepSeek Console` or `Get API Key` action.
- Keep the secure text field and primary `Save` action visible.
- After a successful save, clear the input, collapse the API key section, and refresh balance.
- Show a short success message such as `Saved securely`.

The setup message should be brief enough to preserve the utility feel of the panel.

### Actionable Error States

Error messages should tell the user what to do next.

Recommended behavior:

- Authentication failure: show a clear API key warning and expose `Replace Key` and `Delete Key`.
- Missing key: show setup state instead of a generic failure.
- Network failure: keep the last successful balance if available and offer `Retry`.
- Rate limit or server error: show a short retry-oriented message and offer `Retry`.
- Decoding or unexpected response failure: show refresh failed and offer `Retry`.

To avoid fragile string checks, the core state should expose an error category or failure reason that the view model can use to choose actions.

### API Key Management

The existing security posture remains unchanged: the app never displays a saved API key in plain text.

Recommended behavior:

- Configured state stays collapsed by default.
- Expanding the API key row shows the replacement secure field, `Replace`, and `Delete`.
- `Delete` requires confirmation before removing the key.
- Replace failure keeps the typed input so the user can fix it.
- Replace success clears the input, collapses the row, and shows short success feedback.
- Settings messages must never include the API key or authorization header.

### Refresh Feedback

The refresh button should reflect the current loading state.

Recommended behavior:

- Disable the refresh button while a refresh is already in progress.
- Keep the status line as `Refreshing`.
- Avoid noisy animations in the menu bar or panel.
- Keep the current controller-level overlapping refresh protection.

### Panel Structure

The current panel should remain compact. v0.1.1 may lightly reorder or conditionally expand sections, but it should not become a full settings window.

Recommended behavior:

- In normal loaded state, keep API key management visually secondary.
- In setup or authentication failure states, make API key controls more prominent.
- Keep the footer actions as three compact modules: `AutoStart`, `Console`, and `Quit`.
- Defer deeper AutoStart settings and full Settings UI to a later release unless required by the error recovery work.

## Release And Packaging Design

### Version Consistency

v0.1.1 should use one version source for release packaging.

Recommended behavior:

- Set app `CFBundleShortVersionString` to `0.1.1`.
- Set app `CFBundleVersion` to a matching build number, initially `2`.
- Use a release tag such as `v0.1.1-alpha.1`.
- Use a versioned DMG filename such as `API-Inquiry-v0.1.1-alpha.1.dmg`.
- Keep the existing `API-Inquiry-alpha.dmg` name only if the user intentionally wants a stable generic asset name for GitHub downloads. If both are produced, document which one users should download.

### Checksum

The release process should generate a SHA-256 checksum file next to the DMG.

Recommended output:

```text
dist/API-Inquiry-v0.1.1-alpha.1.dmg
dist/API-Inquiry-v0.1.1-alpha.1.dmg.sha256
```

README and release notes should include a command such as:

```bash
shasum -a 256 -c API-Inquiry-v0.1.1-alpha.1.dmg.sha256
```

### DMG Validation

The packaging scripts should continue to verify the final DMG, not only the staging directory.

Validation should cover:

- `API Inquiry.app` exists in the mounted DMG.
- `API Inquiry.app` is not hidden.
- `Applications` symlink exists.
- The app executable exists.
- The app Info.plist has the expected bundle id and version.
- Ad-hoc code signature verification passes.
- The checksum file validates the produced DMG.

### Development Versus Release Validation

During in-progress development, use fast local launch flows.

Recommended behavior:

- Use scripts such as `Scripts/build-local-app.sh` and local `open` commands for UI testing.
- Do not generate DMGs for every intermediate change.
- Run `swift run APIInquiryCoreTestsRunner` for core behavior.
- Only run full app packaging, DMG generation, checksum generation, and release note checks when preparing the v0.1.1 release candidate.

## Architecture Notes

### Error Classification

The current failed state mostly exposes user-facing text. v0.1.1 should add a small internal classification layer so the UI can choose recovery actions without parsing strings.

Possible model:

- `notConfigured`
- `authenticationFailed`
- `rateLimited`
- `networkUnavailable`
- `serverError`
- `decodingFailed`
- `unknown`

The display message can stay friendly and short, while the category drives actions such as `Retry`, `Replace Key`, or `Delete Key`.

### View Model Responsibilities

`MenuBarBalanceViewModel` should continue to own UI-facing state. It can expose small derived properties for:

- whether refresh is currently disabled
- whether the setup guidance should be shown
- which recovery actions are available
- API key save/delete feedback
- versioned display or diagnostic metadata if needed later

The view should remain mostly declarative and avoid duplicating business rules.

### Packaging Scripts

Packaging scripts should avoid hard-coded version drift. A small shared version file or shared script variables are acceptable if they keep the scripts simple.

The release workflow should remain local-first for now. GitHub Actions release automation can be added later if manual release steps become error-prone.

## Testing Plan

### Core Tests

- Missing API key enters setup state.
- Authentication failure exposes replace/delete recovery actions.
- Network, rate limit, server, and decoding failures expose retry actions.
- Refreshing state disables manual refresh.
- Save API key success clears input and produces success feedback.
- Save API key failure keeps input and shows a safe error.
- Delete API key requires a UI confirmation path and marks the app not configured after confirmation.
- No test exposes or logs a real API key.

### Packaging Tests

- Build local app for quick development validation.
- Run `swift run APIInquiryCoreTestsRunner`.
- For release candidate only, run the full DMG packaging script.
- Mount the generated DMG and verify app visibility.
- Verify Info.plist version and bundle id.
- Verify ad-hoc code signature.
- Verify SHA-256 checksum file.

### Manual Tests

- First launch with no API key.
- Save a real API key locally.
- Confirm balance refresh and menu bar display.
- Trigger manual refresh and confirm button state.
- Test invalid API key and confirm recovery actions.
- Cancel API key deletion and confirm the key remains configured.
- Confirm API key deletion and verify setup state returns.
- Open Console from the footer.
- Toggle AutoStart remains functional.
- Launch the in-progress app through local scripts during development.
- Install from DMG only during release candidate validation.

## Rollout Plan

1. Write and review this design spec in English and Chinese.
2. Create the v0.1.1 implementation plan after spec approval.
3. Implement UI and state-model polish first.
4. Implement packaging/version/checksum changes second.
5. Use local app launch for development validation.
6. Use full DMG packaging only for release candidate validation.
7. Prepare bilingual v0.1.1 release notes.
8. Publish the v0.1.1 alpha release after local and DMG validation pass.
