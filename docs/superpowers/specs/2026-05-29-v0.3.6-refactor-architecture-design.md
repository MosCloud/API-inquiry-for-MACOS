# API Inquiry v0.3.6 Architecture Refactor Design

## Summary

v0.3.6-Refactor turns the current multi-provider implementation from an evolved DeepSeek-first code path into a stable built-in-provider architecture. The refactor keeps the current app behavior and visual UI unchanged while reducing provider-specific branching, unifying ViewModels around `MultiProviderBalanceCoordinator`, splitting the oversized Console view, and preparing the codebase for future built-in provider additions.

## Goals

- Preserve all existing user-visible behavior, UI layout, localization, credential storage behavior, menu bar behavior, refresh cadence, and provider API protocols.
- Keep provider support limited to built-in providers. Do not add user-defined providers or a plugin architecture.
- Centralize provider business metadata so the app has one authoritative source for display name, menu prefix, credential account, homepage URL, detail kind, credential management policy, and access purpose.
- Make built-in provider registration, default provider order, and default primary provider come from one registry/factory rather than separate lists in Core and the App entry point.
- Move AppKit/SwiftUI visual provider metadata into an app-layer visual catalog, separate from Core.
- Make `MenuBarBalanceViewModel` and `UsageConsoleViewModel` coordinator-only production ViewModels.
- Split `UsageConsoleView.swift` into focused SwiftUI components without redesigning the UI.
- Separate value formatting, status text, and tone resolution enough that tests can target each responsibility.
- Refresh configured providers concurrently while preserving per-provider loading, error, and last-snapshot isolation.
- Add lightweight UI tokens and accessibility labels where existing controls rely on scattered constants or color-only semantics.
- Keep and strengthen the existing executable test runner as the refactor safety net.

## Non-Goals

- Do not introduce custom provider configuration, dynamic endpoint schemas, provider plugins, or runtime-loaded providers.
- Do not replace `ProviderID` with a fully dynamic string type in this refactor.
- Do not rewrite the app UI or change the current visual direction.
- Do not change Keychain account values or migrate stored credentials.
- Do not change provider HTTP APIs, response parsing behavior, quota thresholds, or release metadata.
- Do not add third-party architecture, dependency injection, or UI libraries.

## Provider Metadata

Core should expose a richer `ProviderDescriptor` as the business metadata source for built-in providers:

- `id: ProviderID`
- `displayName: String`
- `menuPrefix: String`
- `credentialAccount: String`
- `homepageURL: URL`
- `detailKind: ProviderDetailKind`
- `credentialManagement: ProviderCredentialManagement`
- `accessPurpose: ProviderAccessPurpose`

`ProviderCredentialManagement` should express whether a provider uses app-managed API keys or local external configuration, replacing scattered boolean checks such as `supportsConsoleCredentialManagement`. `ProviderAccessPurpose` should express prepaid balance checks versus plan/quota checks so `UsageConsoleViewModel` does not switch on provider ids for purpose copy.

Concrete provider classes should focus on network request construction, response parsing, and provider-specific error mapping. They should no longer duplicate display names, homepage URLs, menu prefixes, or credential account strings.

An app or core-level built-in provider registry should construct the three current provider instances and their descriptors in the same order. `APIInquiryApp.swift` should not maintain a provider list that can drift from `ProviderCatalog.default`.

## Provider Visuals

The app target should own provider visual metadata through a `ProviderVisualCatalog` or equivalent type. It should map `ProviderID` to:

- header logo asset name
- menu icon asset name
- header logo size
- console logo size
- menu icon size
- fallback visual text

`DeepSeekImages` should be renamed or wrapped by a neutral provider visual helper. Core must not import SwiftUI, AppKit, or image resource details.

## Runtime And ViewModels

`ProviderRuntime` should combine:

- `descriptor: ProviderDescriptor`
- `provider: BalanceProvider`
- `controller: BalanceRefreshController`

`MultiProviderBalanceCoordinator` should expose descriptors, providers, states, credential configuration status, primary provider id, added provider ids, add/remove/primary actions, and refresh operations.

`MenuBarBalanceViewModel` and `UsageConsoleViewModel` should depend on `MultiProviderBalanceCoordinator` only. The legacy single-provider initializer and fields should be removed or reduced to test helpers outside production paths. State reads such as provider display name, active state, credential status, and provider summaries should all come from coordinator-backed data.

## Console UI Split

`UsageConsoleView.swift` should become a page shell that owns section selection and top-level layout. The current UI should be split into focused components:

- `UsageConsoleSection`
- `UsageConsoleNavigationView`
- `UsageConsoleSectionHeader`
- `UsageConsoleHomeSection`
- `ProviderStatusCard`
- `UsageConsoleAPISection`
- `ProviderAPISettingsCard`
- `ConsoleSettingsView`
- `ProviderMetricBox`
- `ProviderBadges`
- `ProviderLogoView`

Child views should receive `APIProviderSummary` values and explicit action closures where possible. They should not all observe the full `UsageConsoleViewModel`.

The split is structural only. It must not change spacing, card hierarchy, button placement, confirmation flows, or provider card information.

The root Console view should continue to own transient API-page state such as currently edited provider ids, removal confirmations, and local feedback. Child views should receive bindings, booleans, summaries, and action closures.

## Formatting And Tone Semantics

Formatting should be separated into focused responsibilities:

- `ProviderValueFormatter`: amount, percentage, quota window, and detail value strings.
- `ProviderStatusFormatter`: status text and badge text derived from `BalanceState` and snapshot kind.
- `ProviderToneResolver`: status tone, amount tone, summary health tone, and threshold logic.

Branching by `ProviderSnapshot` kind is acceptable. Branching by specific provider id should be removed unless the provider has a genuine product-specific display rule that cannot be expressed through descriptor metadata.

## Refresh Flow

`MultiProviderBalanceCoordinator.refreshAddedProviders()` should eventually refresh added providers concurrently. This is a behavior optimization and should land after the metadata, ViewModel, and UI split work has tests in place. Each provider keeps its own `BalanceRefreshController`, and each controller keeps its own anti-reentry protection, loading state, error state, and last snapshot.

The UI should continue to show per-provider loading/error states. One slow or failing provider must not block another provider from completing.

## UI Tokens And Accessibility

The app should introduce lightweight tokens, not a large design system:

- spacing values used by Console and menu surfaces
- card and control corner radius
- status and amount tone color mapping
- compact typography helpers where useful

Icon-only buttons, more menus, provider logo buttons, status badges, menu bar badges, and metric boxes should expose accessible labels or combined accessibility elements. Color-coded state must also have text semantics.

## Testing Requirements

The executable test runner remains the primary automated verification surface. The refactor should update or add tests for:

- provider descriptor completeness and uniqueness
- current behavior characterization for menu titles, provider summaries, primary provider selection, add/remove provider behavior, credential deletion, Codex config target, and failed refresh last-snapshot preservation
- descriptor-backed provider runtimes
- ViewModels using coordinator-only state paths
- API provider summary copy derived from metadata
- formatter/tone resolver behavior after split
- concurrent provider refresh and state isolation
- existing provider network parsing and error mapping

Required verification commands:

```bash
swift run APIInquiryCoreTestsRunner
swift build
```

Manual verification should confirm that Home, API, Settings, menu bar label, menu bar details panel, provider add/remove, API key save/delete, Codex config actions, language switching, and launch-at-login controls remain visually and behaviorally unchanged.

## Acceptance Criteria

- The app builds and the full test runner passes.
- Existing UI behavior and layout are preserved.
- Provider business metadata has one Core source of truth.
- Provider visual metadata has one App-layer source of truth.
- Production ViewModels no longer maintain single-provider and coordinator paths in parallel.
- Compatibility initializers, if kept for tests, wrap a coordinator and do not introduce a second production state path.
- `UsageConsoleView.swift` is split into focused components.
- Added-provider refreshes run concurrently with isolated states.
- Accessibility semantics are improved for icon buttons, badges, provider controls, and metric groups.
- Adding a future built-in provider requires changes only in `ProviderID`, descriptor registration, provider implementation, app visual registration, app initialization, and tests. It should not require new provider-specific branches in Console or menu bar UI.
