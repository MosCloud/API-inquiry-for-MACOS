# API Inquiry v0.3.5 Console Home UI Design

## Summary

v0.3.5 refreshes the Console Home page so it visually belongs to the same product language as the menu bar detail panel while keeping the Console's management-oriented content model. The change focuses on Home only: provider rows become translucent health-tinted units, the top-right status badge uses the same aggregated health tone, and the Console window adopts a more translucent macOS material treatment.

This is a visual and display-model polish release. It does not change provider fetching, credential storage, API key management, provider add/remove behavior, or the API and Settings page layouts.

## Goals

- Keep Console Home as the detailed provider management overview rather than turning it into a larger copy of the menu bar detail panel.
- Align Home provider rows with the detail panel's design language: restrained rounded geometry, translucent layers, semantic health colors, and quiet secondary text.
- Add a Home-specific provider health tone derived from the existing `ProviderAmountTone` thresholds.
- Use the most urgent quota or usage window as the provider row tone when a provider exposes multiple quota windows.
- Sync each Home provider row's background tint and top-right status badge color to that aggregated health tone.
- Give the Console window a translucent macOS material feel closer to the menu bar detail panel.
- Preserve API and Settings page behavior and layout in this release.
- Cover the new display-model tone rules with Core tests.

## Non-Goals

- Do not redesign the API page forms, API key confirmation flows, or Settings page.
- Do not add charts, history, budgets, predictions, or analytics.
- Do not change provider refresh intervals, network parsing, provider ordering, add/remove rules, or primary provider persistence.
- Do not recolor every metric value in Home. The Home row tone and status badge carry the health signal.
- Do not change the menu bar title, menu bar detail panel layout, or menu bar detail panel amount-color thresholds.
- Do not expose API keys or change credential storage.

## Design Direction

The chosen direction is "A+C":

- Keep the current Console Home information architecture: provider list, provider name/logo, API key status, validation status, provider detail, reset/plan details, updated time, and menu bar primary marker.
- Add the health-tinted provider row background from the stronger visual option.
- Keep the right-side status text such as `Available` or `Quota available`, but color the badge using the provider row's aggregated health tone.

This keeps Home concrete and scannable without making it feel like a heavy analytics dashboard.

## Home Health Tone Rules

Home uses a new display-model value on `APIProviderSummary`, such as `healthTone: ProviderAmountTone`.

### Balance Providers

Balance providers use the existing balance amount thresholds:

- `totalBalance >= 50`: `.good`
- `totalBalance >= 10` and `< 50`: `.warning`
- `totalBalance < 10`: `.critical`

### Used Plan Providers

Used-plan providers use the existing usage percentage thresholds:

- `usagePercentage <= 40`: `.good`
- `usagePercentage > 40` and `<= 80`: `.warning`
- `usagePercentage > 80`: `.critical`

### Remaining Quota Providers

Remaining-quota providers calculate a tone for every quota window with the existing remaining percentage thresholds:

- `remainingPercentage >= 60`: `.good`
- `remainingPercentage >= 20` and `< 60`: `.warning`
- `remainingPercentage < 20`: `.critical`

The provider row uses the most urgent window tone. Urgency order is:

```text
critical > warning > good > neutral
```

Example: if Codex has `5h remaining = 18%` and `Weekly remaining = 63%`, the Home provider row and top-right badge use `.critical`.

### Non-Quota States

Health tone only takes over when there is usable balance, used-plan, or remaining-quota data. States without usable quota semantics keep state-level meaning:

- `Not configured`: neutral row and neutral badge.
- `Checking`: neutral or refreshing treatment, without quota health coloring.
- `Invalid` and `Unavailable`: warning/error state treatment rather than quota health coloring.
- Failed states with a preserved last snapshot may continue to show the last snapshot detail, but the row should not look healthy if the active state is an authentication or availability failure.

## Visual Treatment

### Console Window

The Console window should use a transparent/non-opaque AppKit shell and a SwiftUI material background:

- Transparent title bar.
- Full-size content view under the title bar where appropriate.
- Non-opaque window background.
- Root SwiftUI background using a macOS material rather than a solid color.

The implementation should prefer system materials and environment behavior over fixed manual alpha values so the UI can follow dark mode and accessibility settings such as reduced transparency.

### Home Provider Rows

Each Home provider row:

- Keeps an 8px continuous corner radius.
- Uses a translucent base fill.
- Adds a subtle health-colored leading wash or tint.
- Keeps metric text neutral and readable.
- Uses a top-right status badge with the same health tone as the row when the row has quota health semantics.
- Uses neutral or state warning treatment for non-quota states.

### Navigation And Section Header

The top segmented navigation can remain structurally unchanged but should sit on a translucent fill compatible with the new window material. The Home section title and Add Provider menu remain in place.

## Architecture

The quota health decision belongs in Core display formatting, not in SwiftUI string inspection.

Recommended changes:

- Extend `APIProviderSummary` with `healthTone: ProviderAmountTone`.
- Add formatter helpers in `ProviderDisplayFormatter` to calculate provider summary health tone from `BalanceState`.
- Keep existing `ProviderStatusTone` for operational states such as neutral, refreshing, success, and warning.
- Let `UsageConsoleView` map `ProviderAmountTone` to translucent row backgrounds and status badge colors.
- Keep window translucency setup in `UsageConsoleWindowController`.

SwiftUI should not derive health from localized strings such as `Available`, `额度可用`, or `18%`.

## Data Flow

1. Provider refresh logic continues to produce `BalanceState`.
2. `UsageConsoleViewModel.providerSummaries` builds each `APIProviderSummary`.
3. `ProviderDisplayFormatter.summaryHealthTone(for:)` calculates the Home row health tone from the current state.
4. `UsageConsoleView.providerStatusRow(_:)` uses `summary.healthTone` for the Home row background and right-side badge.
5. API and Settings sections keep their current controls and behavior.

## Error Handling

- Missing snapshots produce `.neutral`.
- Loading with no last snapshot produces `.neutral`.
- Loading with a last snapshot may use neutral/refreshing UI rather than presenting stale healthy state as current health.
- Authentication and unavailable failures use state warning treatment.
- Failed states should never expose API keys in feedback or display text.

## Testing

Core tests should cover display-model rules:

- DeepSeek balance boundaries: `50`, `49.99`, `10`, `9.99`.
- Used-plan boundaries: `40`, `40.01`, `80`, `80.01`.
- Remaining-quota aggregation: `[60, 59, 20, 19]` should aggregate to `.critical`.
- Remaining-quota mixed windows: `5h = 18`, `Week = 63` should aggregate to `.critical`.
- Non-quota states: not configured, loading without snapshot, authentication failure, and unavailable failure use neutral or state warning behavior rather than healthy quota tint.
- Provider summaries expose the new tone for single-provider and multi-provider Console modes.

Manual verification should confirm:

- Console Home visually uses the same translucent language as the menu bar detail panel.
- Provider row background and top-right status badge change together.
- Home remains a detailed provider overview rather than a hero-style detail page.
- API and Settings page flows still work and are not redesigned by this release.
