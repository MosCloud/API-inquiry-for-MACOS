# API Inquiry v0.3.4 Detail Number Color Design

## Summary

v0.3.4 improves the expanded menu detail panel by coloring only the primary module's large quota or balance numbers according to quota health. The change helps users scan whether the current primary provider is healthy, close to a warning threshold, or urgent without changing any provider data, menu bar title text, status text, secondary rows, or Console UI.

## Goals

- Color the large number in the primary detail module for balance, used-plan, and remaining-quota providers.
- Keep the existing green tone used by the available status as the good color reference.
- Use yellow for normal warning states and red for urgent states.
- Keep the existing menu bar title, status text color, status dot color, secondary provider list, and Console page unchanged.
- Cover threshold boundaries with automated tests in the Core display model.

## Non-Goals

- Do not change menu bar title rendering or menu bar icon text.
- Do not change provider status text, status tone, or status dot behavior.
- Do not color secondary provider row numbers.
- Do not change Console Home, Console API, or any other Console number rendering.
- Do not add notifications, alerts, budget forecasts, or historical charts.
- Do not change provider fetch logic, API parsing, refresh cadence, credentials, or persistence.

## Color Rules

The color rules apply only to values shown as large numbers in the expanded menu's primary module.

### Balance Providers

DeepSeek-style recharge balance providers use the total balance amount.

- `totalBalance >= 50`: good, rendered green.
- `totalBalance >= 10` and `< 50`: normal warning, rendered yellow.
- `totalBalance < 10`: urgent, rendered red.

Boundary behavior favors the healthier tier when the value equals a boundary. For example, `50` is green and `10` is yellow.

### Used-Plan Providers

Zhipu GLM Coding Plan-style providers use the used percentage.

- `usagePercentage <= 40`: good, rendered green.
- `usagePercentage > 40` and `<= 80`: normal warning, rendered yellow.
- `usagePercentage > 80`: urgent, rendered red.

Boundary behavior favors the healthier tier when the value equals a boundary. For example, `40` is green and `80` is yellow.

### Remaining-Quota Providers

OpenAI/Codex-style providers use the remaining percentage for each displayed quota window.

- `remainingPercentage >= 60`: good, rendered green.
- `remainingPercentage >= 20` and `< 60`: normal warning, rendered yellow.
- `remainingPercentage < 20`: urgent, rendered red.

Boundary behavior favors the healthier tier when the value equals a boundary. For example, `60` is green and `20` is yellow.

## Architecture

Add a small Core display tone type for primary detail numbers, with cases for good, warning, urgent, and neutral fallback if needed for placeholder values. Core owns the threshold logic because the thresholds describe provider quota semantics rather than SwiftUI styling.

Extend `PrimaryProviderDisplayParts` with an amount tone for the single large-number module used by balance and used-plan providers. Extend `QuotaWindowDisplayRow` with an amount tone for the OpenAI/Codex primary quota rows, because each row can have a different remaining percentage and therefore a different color.

`MenuBarContentView` maps the Core tone to SwiftUI colors:

- good -> `.green`
- warning -> `.yellow`
- urgent -> `.red`
- neutral -> `.primary`

The view applies the mapped color only to:

- `balanceAmountLine(parts:amountSize:)` for `Text(parts.amountText)`
- `quotaHeroRows` for `Text(row.amountText)`

All other text keeps its existing foreground style.

## Data Flow

1. Providers continue returning the existing `ProviderSnapshot` values.
2. `ProviderDisplayFormatter.primaryDisplayParts(...)` calculates the amount tone for balance and used-plan snapshots.
3. `MenuBarBalanceViewModel.primaryQuotaWindowRows` calculates the amount tone for each remaining-quota window through Core formatting helpers.
4. `MenuBarContentView` renders only the primary module amount text with the mapped color.

## Error Handling And Fallbacks

Placeholder values such as `--`, missing snapshots, loading without a last snapshot, and not-configured states use the neutral tone and render as `.primary`. Failed states that preserve a last snapshot keep using that last snapshot's calculated amount tone for the primary module number.

## Testing

Core tests should verify the threshold logic without needing SwiftUI inspection:

- DeepSeek balance tone boundaries: `50`, `49.99`, `10`, and `9.99`.
- Zhipu used-plan tone boundaries: `40`, `40.01`, `80`, and `80.01`.
- OpenAI/Codex remaining-quota boundaries: `60`, `59`, `20`, and `19`.
- Primary display parts expose the expected tone for balance and used-plan snapshots.
- Quota window display rows expose the expected tone per row.
- Secondary provider rows are not extended or rendered with the primary amount tone.

Manual verification should run the local menu bar app after tests and confirm that only the expanded detail panel's primary large numbers change color.
