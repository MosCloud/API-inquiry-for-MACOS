# API Inquiry v0.3.6 API Page UI Design

## Summary

v0.3.6 refines the Console API page from tall provider cards with equally weighted buttons into compact provider access-management modules. Each provider remains an independent card, the left-side provider identity reuses the Home page provider chip, and the right side focuses on follow-up management actions. Home remains the primary place to add providers and start first-time setup; the API page manages credentials, local configuration, and provider removal afterward.

## Goals

- Keep each provider as an independent card module instead of merging providers into a table or a single list container.
- Make cards much shorter than Home provider rows by using a single-line primary information structure.
- Reuse the Home page provider chip for the left-side provider display: logo/name/external arrow.
- Place the status area in the middle column, left-aligned inside that column rather than right-aligned near actions.
- Show DeepSeek as `Configured` with `Available for prepaid balance checks`.
- Show Zhipu BigModel as `Configured` with `Available for plan balance checks`.
- Show OpenAI/Codex as `Loaded` with `Available for plan balance checks`.
- Remove explanatory API-page copy about Keychain.
- Use the DeepSeek/Zhipu primary button for configuring or replacing API keys.
- Use `Open Config` as the OpenAI/Codex primary button, opening the local Codex `auth.json` or config directory.
- Move destructive and low-frequency actions, such as deleting keys and removing providers, into a more menu while preserving confirmation.
- Ensure the delete-key confirmation no longer mentions Keychain; it should only describe deleting the saved API key.

## Non-Goals

- Do not change Home provider overview, health tinting, or the primary add-provider entry point.
- Do not change provider refresh cadence, balance/plan query protocols, or credential storage strategy.
- Do not redesign the navigation bar, Settings page, or menu-bar detail panel.
- Do not repeat safety, privacy, or Keychain explanations already covered by the download page.
- Do not expose any API key or Codex token.

## Page Roles

Home is the discovery, add-provider, and first-access entry point. After adding a provider from Home, the user can land on the API page to complete first-time key entry. The API page is the follow-up access-management page: it shows access status, configures or replaces keys, deletes keys, removes providers, and opens Codex local configuration.

## API Card Layout

Each provider card uses a three-zone single-line layout:

```text
[provider chip ↗]      [status] purpose copy                 [primary button] [...]
```

- Left column: reuse the Home provider chip style and external-open behavior.
- Middle column: status pill plus purpose copy, left-aligned within the middle column.
- Right column: primary action button and more menu, right-aligned.

Recommended card height is 76-88 pt with 10-12 pt card spacing. Backgrounds, corner radius, and translucent hierarchy should continue the Home page's quiet 8 pt continuous-radius material language.

## Status And Actions

| Provider Type | Status | Purpose Copy | Primary Action |
| --- | --- | --- | --- |
| Prepaid balance API, such as DeepSeek | Configured / Not configured | Available for prepaid balance checks | Configure Key / Replace Key |
| Plan balance API with app-managed key, such as Zhipu | Configured / Not configured | Available for plan balance checks | Configure Key / Replace Key |
| Plan balance API loaded from local config, such as OpenAI/Codex | Loaded / Not loaded | Available for plan balance checks | Open Config |

More menu:

- Providers with app-managed keys: Delete API Key, Remove Provider.
- OpenAI/Codex: Show Config in Finder, Remove Provider.

The left provider chip owns the external provider page action, so the primary button and more menu should not duplicate that action.

## OpenAI/Codex Config

OpenAI/Codex continues to use the existing `CodexCredentialStore` to read credentials from local configuration. The API page needs one openable target:

- If a usable `auth.json` exists, `Open Config` opens that file.
- If `auth.json` does not exist, open the preferred config directory, such as `$CODEX_HOME` or `~/.codex`.
- If no config directory can be resolved, show non-sensitive feedback and never display a token.

Config target resolution belongs in Core/security or ViewModel display model code. SwiftUI should not duplicate the environment-variable lookup logic.

## Interaction States

- Unconfigured providers with app-managed keys expand a compact input state inside the card, keeping the existing save/cancel flow.
- Replacing a key shows only a new-key input and never displays the old key.
- Deleting a key and removing a provider keep a second confirmation, compactly expanded below the card row.
- Delete-key confirmation describes the user action only and does not repeat implementation details about storage.
- Removing a provider remains disabled when only one provider is present.

## Testing

- Core tests cover API-page status/purpose copy: DeepSeek prepaid balance, Zhipu plan balance, and Codex loaded/not-loaded.
- Core tests cover Codex config URL selection: existing auth file returns the file; missing auth file returns the config directory.
- Localization tests cover delete API key confirmation copy without Keychain wording.
- Swift build verifies SwiftUI compilation.
- Manual visual verification confirms independent compact cards, Home-consistent left chip, left-aligned middle status, and non-overlapping right actions.
