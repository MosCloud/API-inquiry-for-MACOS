# API Inquiry v0.3.0 Multi-Provider Design

## Summary

v0.3.0 upgrades API Inquiry from a DeepSeek-only balance tool into a multi-provider status tool. The first real providers are DeepSeek and Zhipu GLM Coding Plan.

The menu bar stays restrained: it shows only the high-frequency status for one Primary Provider. Opening the menu bar details panel shows all added providers. Provider onboarding, API key management, and Primary Provider selection remain in the Console.

## Goals

- Support a built-in provider catalog with DeepSeek and Zhipu GLM Coding Plan as the first providers.
- Preserve DeepSeek's existing Keychain account, refresh behavior, balance display, and upgrade compatibility.
- Add real plan usage querying for Zhipu GLM Coding Plan.
- Show only the Primary Provider in the menu bar.
- Show all provider details in the menu bar details panel.
- Keep the Console as the provider management surface for API keys, adding providers, and selecting the Primary Provider.
- Keep each provider's API key, refresh state, last snapshot, and error state isolated.
- Cover multi-provider state isolation, Keychain account isolation, Console management, menu bar display, and Zhipu parsing with automated tests.

## Non-Goals

- No arbitrary custom providers, dynamic endpoint configuration, scripting plugins, or provider marketplace.
- No historical usage import, usage charts, cost analysis, or budget alerts.
- No cross-currency balance aggregation, exchange rates, or multi-provider total.
- No multiple accounts or multiple API keys for the same provider.
- No cloud sync, account system, team sharing, or cross-device state sync.
- No web-console scraping or browser automation.
- No Developer ID signing or Apple notarization.
- The menu bar details panel does not become a management surface; management remains in the Console.

## Provider Scope

### DeepSeek

DeepSeek remains the default provider and default Primary Provider.

DeepSeek displays a balance:

- Menu bar: compact balance, for example `¥68.6`.
- Top details area: existing large DeepSeek balance treatment, for example `¥68.65 CNY`.
- Secondary row: provider name, balance, availability, and last refresh time.

DeepSeek must keep using the current Keychain account:

```text
deepseek-api-key
```

### Zhipu GLM Coding Plan

Zhipu GLM Coding Plan is v0.3.0's first new real provider.

Zhipu displays plan usage rather than cash balance:

- Menu bar: window and usage percentage, for example `5h 17%`.
- Top details area: when Zhipu is the Primary Provider, show `5h` and `17% used`, with only the number `17` enlarged.
- Secondary row: provider name, window, usage percentage, availability, next reset time, and last refresh time.

Official Zhipu documentation confirms:

- GLM Coding Plan uses the dedicated Coding endpoint: `https://open.bigmodel.cn/api/coding/paas/v4`.
- The API uses HTTP Bearer authentication: `Authorization: Bearer YOUR_API_KEY`.
- Error codes include GLM Coding Plan states such as expired plan `1309`, and usage limit or reset states such as `1308` and `1310`.

The quota endpoint must be locally validated before implementation. Public references commonly mention these candidate endpoints:

```text
https://open.bigmodel.cn/api/monitor/usage/quota/limit
https://bigmodel.cn/api/monitor/usage/quota/limit
```

Validation may only use local input or the real key saved in Keychain. Real API keys must never be written into chat, docs, source code, logs, or test fixtures.

## Primary Provider Rules

The Primary Provider is the first provider shown in the menu bar.

- DeepSeek is the default Primary Provider.
- The Console provides a `Show in Menu Bar` action or equivalent to set the Primary Provider.
- The menu bar label shows only the Primary Provider's adaptive detail.
- If the Primary Provider has no configured key, the menu bar shows that provider's setup state.
- If the Primary Provider refresh fails but has a previous snapshot, the menu bar keeps showing the previous usable snapshot and the details panel shows the error state.
- The details panel always shows the Primary Provider at the top.
- The details panel shows other added providers as compact rows below.

## Menu Bar Details Panel

### Top Primary Provider Area

The top area preserves the current DeepSeek details hierarchy: provider identity, primary number, status, last refresh time, and quick actions.

Balance providers:

- The primary number is the balance.
- Currency symbol and currency code have weaker visual weight.
- Example: `¥68.65 CNY`.

Plan providers:

- The primary number is the numeric part of the usage percentage.
- Window and unit are supporting information.
- Example: `5h`, `17% used`, with `17` as the largest text.

### Other Provider Rows

Each non-primary provider occupies one row.

Each row shows:

- Provider name.
- Adaptive detail: balance or plan usage.
- Availability/status, such as `Available`, `Balance insufficient`, `Plan available`, `Limit reached`, `Invalid`, `Unavailable`, or `Not configured`.
- Last refresh time.
- Optional next reset time for plan providers.

The details panel may provide refresh and Console buttons, but it does not provide API key editing, provider deletion, or Add Provider flows.

## Console Management

The Console keeps the `Home` and `API` pages.

### Home

Home shows the added providers.

Each provider summary shows:

- Name.
- API key status.
- Validation status.
- Balance or plan usage.
- Last refresh time.
- Whether it is the Primary Provider.

`Add Provider` opens a built-in provider picker that only shows built-in providers that have not been added yet.

### API

The API page shows per-provider management panels.

Each panel supports:

- Save API key.
- Replace API key.
- Delete API key with inline confirmation.
- Set as Primary Provider.
- Remove provider from the added provider list.

When removing a provider, if it has a saved API key, the app must clearly state that the key will also be deleted. After confirmation, the app deletes that provider's Keychain item and removes the provider from the local added list.

DeepSeek must never disappear from the built-in catalog. Even if the DeepSeek API key is deleted, DeepSeek remains available as an addable or default provider for upgrade compatibility and fallback.

## Architecture

### Provider Catalog

Add a provider catalog to centralize built-in providers, default ordering, and the default Primary Provider.

Each provider descriptor includes:

- `ProviderID`
- display name
- menu prefix
- credential account
- homepage URL
- detail kind: balance or plan usage
- provider factory

ProviderID in v0.3.0 includes at least:

```swift
case deepseek
case zhipuCodingPlan
```

### Single-Provider Controller

`BalanceRefreshController` remains a single-provider state machine instead of becoming a multi-provider controller.

This preserves existing tested behavior:

- One provider.
- One credential account.
- One `BalanceState`.
- One auto-refresh task.
- Preserve that provider's previous snapshot on failure.
- Prevent overlapping refreshes for the same provider.

### Multi-Provider Coordinator

Add a multi-provider coordinator or runtime collection that composes multiple single-provider controllers.

Responsibilities:

- Get a provider runtime by `ProviderID`.
- Refresh one provider.
- Refresh all configured providers.
- Start and stop auto-refresh for all providers.
- Expose state and last snapshot for each provider.

The coordinator does not store plain API keys and does not parse UI copy.

### Provider Detail Model

Add a unified provider detail display model for balance and plan usage details.

Balance detail includes:

- total balance
- currency
- availability
- fetched at

Plan usage detail includes:

- window label, for example `5h`
- usage percentage, for example `17`
- usage text, for example `17% used`
- reset time
- availability
- fetched at

Menu bar, details panel, and Console summaries should derive their display from this model rather than reverse-engineering colors or layout from status strings.

### Local Preferences

API keys are only stored in Keychain. Non-sensitive preferences may be stored locally, for example in UserDefaults.

Allowed local preferences:

- Added provider IDs.
- Primary Provider ID.

Forbidden local preferences:

- API keys.
- Authorization headers.
- Sensitive fields from API responses.

## Error Handling

All providers share these base states:

- `Not configured`
- `Checking`
- `Available`
- `Invalid`
- `Unavailable`

Balance providers also support:

- `Balance insufficient`

Plan providers also support:

- `Plan available`
- `Limit reached`
- `Plan expired`

Suggested Zhipu error mapping:

- HTTP 401 or business authentication errors: `Invalid`
- HTTP 429 and business errors `1302`, `1303`, `1305`: `Unavailable` or `Limit reached`, depending on response semantics
- Business errors `1308`, `1310`: `Limit reached`; show reset time when available
- Business error `1309`: `Plan expired`
- 5xx: `Unavailable`
- Decoding failure: `Unavailable`, preserving the previous snapshot

If saving an API key succeeds but refresh fails, keep the current behavior: save the key, show warning feedback, and mark the provider `Invalid` or `Unavailable`.

Error feedback must never include API keys, Authorization headers, or sensitive raw responses.

## Testing

Automated tests use the existing `APIInquiryCoreTestsRunner`.

Add or update coverage for:

- Provider catalog default order, unique IDs, unique credential accounts, and unique menu prefixes.
- DeepSeek still uses `deepseek-api-key`.
- Zhipu provider request URL, HTTP method, Authorization header, and response parsing.
- Zhipu plan usage snapshot parsing, including window, percentage, reset time, and availability.
- Zhipu error mapping, including authentication failure, rate limit, usage limit, expired plan, and server errors.
- Multi-provider credential isolation: saving, replacing, or deleting one key does not affect another provider.
- Multi-provider runtime isolation: one provider refresh failure does not pollute another provider's state or last snapshot.
- Console summaries show multiple providers.
- Console per-provider API key save, replace, delete, and feedback.
- Primary Provider selection and restoration.
- Menu bar label formatting for balance and plan providers.
- Menu bar details hero model and secondary rows model.
- API keys never appear in feedback, logs, or test failure messages.

Verification commands:

```bash
swift run APIInquiryCoreTestsRunner
swift build
Scripts/build-local-app.sh
```

## Manual Checks

- Launch with no keys: DeepSeek is the default Primary Provider and the menu bar shows setup state.
- Console can add Zhipu GLM Coding Plan.
- After saving a DeepSeek key, the menu bar and details panel show balance.
- After saving a Zhipu key, the Console and details panel show plan usage.
- After setting Zhipu as Primary Provider, the menu bar shows plan usage such as `5h 17%`.
- When Zhipu is Primary Provider, the details panel enlarges `17` and shows `5h` and `% used`.
- When DeepSeek is Primary Provider, the details panel keeps the current balance treatment.
- Non-primary providers appear as one-row status entries in the details panel.
- Deleting the Zhipu key does not affect the DeepSeek key, balance, or menu bar state.
- Zhipu limit reached, expired plan, or invalid key states show accurate copy without exposing the key.
