# API Inquiry v0.3.6 架构重构设计

## 摘要

v0.3.6-Refactor 的目标，是把当前已经可用的多 Provider 实现，从“由 DeepSeek 单 Provider 逐步演进出来的代码路径”，整理成稳定的内置 Provider 架构。重构需要保持当前 app 功能和视觉 UI 不变，同时降低 provider-specific 分支扩散，统一 ViewModel 到 `MultiProviderBalanceCoordinator`，拆分过大的 Console View，并为后续新增内置 Provider 预留维护空间。

## 目标

- 保持所有用户可见行为、UI 布局、本地化、凭证存储、菜单栏行为、刷新频率和 provider API 协议不变。
- Provider 支持范围仍限定为内置 Provider；不加入自定义 Provider 或插件架构。
- 集中管理 provider 业务元数据，使 display name、menu prefix、credential account、homepage URL、detail kind、凭证管理策略和访问用途有唯一来源。
- 让内置 provider 注册、默认 provider 顺序和默认 primary provider 来自同一个 registry/factory，避免 Core 和 App 入口维护两份列表。
- 将 AppKit/SwiftUI 相关的 provider 视觉元数据放到 App 层 visual catalog，与 Core 分离。
- 让 `MenuBarBalanceViewModel` 和 `UsageConsoleViewModel` 的生产路径只依赖 coordinator。
- 将 `UsageConsoleView.swift` 拆成聚焦的 SwiftUI 组件，但不重新设计 UI。
- 适度拆分数值格式化、状态文案和 tone 判断，让测试可以覆盖更小职责。
- 将已添加 provider 的刷新改为并发，同时保持每个 provider 的 loading、error 和 last snapshot 状态隔离。
- 建立轻量 UI token，并为依赖图标或颜色表达的控件补充可访问性语义。
- 保留并增强现有 executable test runner 作为重构保护网。

## 非目标

- 不引入自定义 Provider 配置、动态 endpoint schema、Provider 插件或运行时加载 Provider。
- 本轮重构不把 `ProviderID` 改成完全动态的 string 类型。
- 不重写 app UI，不改变当前视觉方向。
- 不修改 Keychain account 值，不迁移已保存凭证。
- 不改变 provider HTTP API、响应解析行为、额度阈值或发布元数据。
- 不引入第三方架构、依赖注入或 UI 库。

## Provider 元数据

Core 应扩展 `ProviderDescriptor`，让它成为内置 Provider 的业务元数据中心：

- `id: ProviderID`
- `displayName: String`
- `menuPrefix: String`
- `credentialAccount: String`
- `homepageURL: URL`
- `detailKind: ProviderDetailKind`
- `credentialManagement: ProviderCredentialManagement`
- `accessPurpose: ProviderAccessPurpose`

`ProviderCredentialManagement` 用于表达 provider 是使用 app 管理的 API Key，还是读取本地外部配置，替代分散的 `supportsConsoleCredentialManagement` 布尔判断。`ProviderAccessPurpose` 用于表达充值余额查询和套餐/额度查询，避免 `UsageConsoleViewModel` 为访问用途文案 switch provider id。

具体 provider 类只负责网络请求构造、响应解析和 provider-specific 错误映射，不再重复声明 display name、homepage URL、menu prefix 或 credential account。

App 或 Core 层应有一个内置 provider registry，用同一顺序构造当前三个 provider 实例和 descriptor。`APIInquiryApp.swift` 不应再维护一份可能与 `ProviderCatalog.default` 漂移的 provider 列表。

## Provider 视觉信息

App target 应通过 `ProviderVisualCatalog` 或等价类型管理 provider 视觉元数据。它按 `ProviderID` 映射：

- header logo asset name
- menu icon asset name
- header logo size
- console logo size
- menu icon size
- fallback visual text

`DeepSeekImages` 应改名或包装为中性的 provider visual helper。Core 不能引入 SwiftUI、AppKit 或图片资源细节。

## Runtime 与 ViewModel

`ProviderRuntime` 应组合：

- `descriptor: ProviderDescriptor`
- `provider: BalanceProvider`
- `controller: BalanceRefreshController`

`MultiProviderBalanceCoordinator` 应暴露 descriptor、provider、state、凭证配置状态、primary provider id、已添加 provider id、添加/删除/设为主 provider 操作和刷新操作。

`MenuBarBalanceViewModel` 与 `UsageConsoleViewModel` 应只依赖 `MultiProviderBalanceCoordinator`。历史 single-provider initializer 和字段应删除，或降级为生产路径之外的测试辅助。provider display name、active state、credential status、provider summaries 等状态读取都应来自 coordinator-backed 数据。

## Console UI 拆分

`UsageConsoleView.swift` 应成为页面外壳，只保留 section 选择和顶层布局。当前 UI 拆分为聚焦组件：

- `UsageConsoleSection`
- `UsageConsoleNavigationView`
- `UsageConsoleSectionHeader`
- `UsageConsoleHomeSection`
- `ProviderStatusCard`
- `UsageConsoleAPISection`
- `ProviderAPISettingsCard`
- `ConsoleSettingsView`
- `ProviderMetricBox`
- `ProviderBadges`
- `ProviderLogoView`

子 View 应尽量接收 `APIProviderSummary` 和明确的 action closure，不应全部直接观察完整 `UsageConsoleViewModel`。

拆分只做结构调整，不改变 spacing、卡片层级、按钮位置、确认流程或 provider 卡片信息。

Console 根 View 继续持有 API 页面临时状态，例如正在编辑的 provider id、移除确认、本地 feedback。子 View 接收 binding、布尔状态、summary 和 action closure。

## 格式化与状态语义

格式化逻辑拆分为更聚焦的职责：

- `ProviderValueFormatter`：金额、百分比、quota window 和 detail value 文本。
- `ProviderStatusFormatter`：由 `BalanceState` 和 snapshot kind 推导状态文案与 badge 文案。
- `ProviderToneResolver`：status tone、amount tone、summary health tone 和阈值逻辑。

基于 `ProviderSnapshot` 类型的分支是合理的。除非存在无法通过 descriptor 表达的产品特定展示规则，否则应移除基于具体 provider id 的分支。

## 刷新流程

`MultiProviderBalanceCoordinator.refreshAddedProviders()` 最终应并发刷新已添加 provider。这是行为优化，应在 metadata、ViewModel 和 UI 拆分已有测试保护后落地。每个 provider 继续拥有自己的 `BalanceRefreshController`，每个 controller 自己负责防重入、loading state、error state 和 last snapshot。

UI 继续显示每个 provider 独立的 loading/error 状态。一个 provider 慢或失败，不能阻塞另一个 provider 完成刷新。

## UI Token 与可访问性

本轮只引入轻量 token，不做大型设计系统：

- Console 和菜单面板使用的 spacing
- card/control corner radius
- status 和 amount tone 到颜色的映射
- 必要时增加 compact typography helper

纯图标按钮、更多菜单、provider logo 按钮、状态 badge、菜单栏 badge 和 metric box 应提供 accessibility label 或合并后的 accessibility element。用颜色表达的状态必须同时有文字语义。

## 测试要求

现有 executable test runner 仍是主要自动化验证面。重构需要更新或新增测试覆盖：

- provider descriptor 完整性和唯一性
- 当前行为 characterization：菜单标题、provider summary、primary provider 选择、添加/移除 provider、凭证删除、Codex config target 和失败刷新保留 last snapshot
- descriptor-backed provider runtime
- ViewModel 只使用 coordinator state path
- API provider summary 文案由 metadata 推导
- formatter/tone resolver 拆分后的行为
- provider 并发刷新和状态隔离
- 现有 provider 网络解析与错误映射

必要验证命令：

```bash
swift run APIInquiryCoreTestsRunner
swift build
```

手动验证需要确认 Home、API、Settings、菜单栏 label、菜单栏详情面板、provider 添加/删除、API key 保存/删除、Codex 配置操作、语言切换和开机启动控制在视觉和行为上保持不变。

## 验收标准

- app 可以 build，完整 test runner 通过。
- 现有 UI 行为和布局保持不变。
- Provider 业务元数据在 Core 中有唯一来源。
- Provider 视觉元数据在 App 层有唯一来源。
- 生产 ViewModel 不再并行维护 single-provider 与 coordinator 两套路由。
- 如果为了测试保留兼容 initializer，它必须包装 coordinator，不能引入第二条生产状态路径。
- `UsageConsoleView.swift` 被拆分为聚焦组件。
- 已添加 provider 的刷新并发执行，且状态隔离。
- 图标按钮、badge、provider 控件和 metric group 的可访问性语义得到增强。
- 后续新增内置 Provider 时，只需修改 `ProviderID`、descriptor 注册、provider 实现、App visual 注册、app 初始化和测试；不应再需要在 Console 或菜单栏 UI 中添加 provider-specific 分支。
