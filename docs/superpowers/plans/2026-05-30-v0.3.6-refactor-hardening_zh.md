# v0.3.6-Refactor Runtime Hardening Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在不改变现有功能和 UI 的前提下，收紧本轮重构审阅中指出的 provider runtime 边界语义。

**Architecture:** 保持 `ProviderRegistration -> ProviderRuntime -> ViewModel` 的现有方向不变，只对默认 provider 语义和 credential-management 防御性 fallback 做小范围强化。生产入口继续显式传入 `BuiltInProviderRegistry.default.registrations` 与 registry default provider。

**Tech Stack:** Swift Package Manager, Swift Concurrency, Combine, local `APIInquiryCoreTestsRunner`.

---

## 工作边界

- 不改 SwiftUI 视觉结构、布局、文案和用户可见流程。
- 不引入自定义 provider，也不把 `ProviderID` 改成动态 string。
- 不改 Keychain 存储格式、不迁移用户偏好。
- 不重构 Console UI、不重写 formatter。

## Task 1: 收紧 coordinator 默认 provider 语义

**Files:**
- Modify: `Sources/APIInquiryCore/Refresh/MultiProviderBalanceCoordinator.swift`
- Test: `Sources/APIInquiryCoreTestsRunner/MultiProviderBalanceCoordinatorTests.swift`

- [x] **Step 1: 写 RED 测试**

在 `MultiProviderBalanceCoordinatorTests.run(using:)` 中新增测试入口：

```swift
testCoordinatorCanInferDefaultProviderFromRegistrations(using: harness)
```

新增测试方法，显式传入 `defaultProviderID: nil`，表达“如果调用方使用自定义 registration 集合且没有提供 default provider，coordinator 应从 registration 顺序中选择第一个 provider”：

```swift
@MainActor
private static func testCoordinatorCanInferDefaultProviderFromRegistrations(using harness: TestHarness) {
    let zhipu = MockBalanceProvider(id: .zhipuCodingPlan, results: [])
    let coordinator = MultiProviderBalanceCoordinator(
        registrations: [testRegistration(for: zhipu)],
        credentialStore: InMemoryCredentialStore(),
        preferences: InMemoryProviderPreferencesStore(
            addedProviderIDs: [],
            primaryProviderID: nil
        ),
        defaultProviderID: nil
    )

    harness.expectEqual(coordinator.addedProviderIDs, [.zhipuCodingPlan], "coordinator infers added provider from registrations")
    harness.expectEqual(coordinator.primaryProviderID, .zhipuCodingPlan, "coordinator infers default provider from registrations")
}
```

- [x] **Step 2: 运行测试确认失败**

Run:

```bash
swift run APIInquiryCoreTestsRunner
```

Expected: 编译失败，因为当前 `defaultProviderID` 参数类型是 `ProviderID`，不能传 `nil`。

- [x] **Step 3: 写最小实现**

把 `MultiProviderBalanceCoordinator.init` 的参数从：

```swift
defaultProviderID: ProviderID = BuiltInProviderRegistry.default.defaultProviderID,
```

改为：

```swift
defaultProviderID: ProviderID? = nil,
```

并在初始化时解析默认 provider：

```swift
guard let resolvedDefaultProviderID = defaultProviderID ?? registrations.first?.descriptor.id else {
    preconditionFailure("At least one provider registration is required.")
}
```

构建 `runtimesByProviderID` 前先用 descriptor id 校验 default provider 已注册，避免未注册 default 触发 provider factory：

```swift
let providerOrder = registrations.map(\.descriptor.id)
precondition(
    providerOrder.contains(resolvedDefaultProviderID),
    "Default provider ID \(resolvedDefaultProviderID.rawValue) must be registered."
)
self.defaultProviderID = resolvedDefaultProviderID
```

保留现有生产调用方显式传入 `registry.defaultProviderID` 的行为。

- [x] **Step 4: 运行测试确认通过**

Run:

```bash
swift run APIInquiryCoreTestsRunner
```

Expected: PASS，且 expectation 数量增加。

## Task 2: credential-management 缺 descriptor 时默认拒绝

**Files:**
- Modify: `Sources/APIInquiryCore/ViewModels/UsageConsoleViewModel.swift`

- [x] **Step 1: 修改防御性 fallback**

把：

```swift
coordinator.descriptor(for: id)?.credentialManagement.supportsConsoleCredentialManagement ?? true
```

改为：

```swift
coordinator.descriptor(for: id)?.credentialManagement.supportsConsoleCredentialManagement ?? false
```

说明：正常 runtime 下 descriptor 不会缺失；如果异常缺失，UI 不应乐观允许保存/删除 key。

- [x] **Step 2: 运行现有 ViewModel 测试**

Run:

```bash
swift run APIInquiryCoreTestsRunner
```

Expected: PASS。该 fallback 是异常防御路径，当前 coordinator public API 下缺 descriptor 与已配置 credential 不会同时成立，因此不强行为不可达状态增加测试 seam。
