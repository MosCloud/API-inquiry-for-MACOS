# API Inquiry v0.3.7 UI Polish Plan

> **For agentic workers:** Implement this plan in small reviewable steps. Preserve the current product style and visible workflows unless a step explicitly says otherwise.

**Goal:** Polish the app UI before the v0.4.0 provider expansion, while keeping API Inquiry minimal, restrained, efficient, and lightly technical.

**Design principle:** UI must serve the content. Remove or reject any animation, decoration, layout change, or visual treatment that makes status, quota, provider, or credential information harder to scan.

**Architecture:** Keep the v0.3.6-Refactor provider architecture intact. UI changes should mainly stay in the App target, with Core changes limited to display state that removes fragile string-based UI decisions or improves accessibility semantics.

**Technical baseline:** Swift 5.9, macOS 13+, SwiftUI/AppKit, local `APIInquiryCoreTestsRunner`.

---

## Style Guardrails

- Keep the overall feel minimal, calm, native macOS, and lightly technical.
- Treat the menu bar panel as a quick status surface, not a dashboard.
- Treat Console as a focused management surface, not an operations backend.
- Use motion only for state feedback: loading, successful refresh, errors, and small value changes.
- Avoid decorative motion, heavy glow, large gradients, ornamental shadows, or visual elements that compete with content.
- Use status color sparingly and semantically.
- Keep provider brand visuals recognizable but compact.
- Do not add new third-party UI or architecture dependencies.

## Scope

### 1. Microinteractions And State Feedback

- Add a restrained loading animation to refresh controls.
- Add subtle transitions for error and recovery action appearance.
- Animate state tone changes where the current UI already uses status color.
- Gate macOS 14-only numeric transitions behind availability checks; macOS 13 must keep a safe fallback.

### 2. UI Semantics And Robustness

- Replace fragile string comparison in API access badges with explicit display state.
- Improve help/accessibility copy for menu and icon buttons, especially "more actions" controls.
- Avoid UI decisions that depend on localized strings.

### 3. Color And Accessibility Polish

- Centralize light/dark-safe tone colors in the existing token layer.
- Improve warning color readability without making the palette louder.
- Add concise accessibility labels/values for provider rows, quota hero rows, status badges, and icon buttons.
- Preserve visible text for color-coded states so color is not the only signal.

### 4. Menu Bar Maintainability

- Split `MenuBarContentView` into focused views without changing visual behavior first.
- Suggested components:
  - `MenuBarHeaderView`
  - `MenuBarQuotaHeroView`
  - `MenuBarStatusView`
  - `MenuBarSecondaryProvidersView`
  - `MenuBarFooterView`
- Keep the 320 pt panel width unless a specific layout problem is proven during review.

### 5. Console Home Light Hierarchy

- Improve scanability only where it serves the content.
- Consider moving low-priority update time out of the main metric row if density becomes high.
- Consider a very restrained leading status indicator only if it improves scan speed without adding visual noise.
- Do not make provider cards look like a dashboard.

### 6. Console API Interaction Polish

- Keep current single-row provider management structure.
- Add focus handling for API key editing.
- Rename generic menu help/accessibility copy from provider removal to "more actions".
- Treat popover/sheet confirmation as optional and only after manual review, because it changes the interaction model.

### 7. UI Code Hygiene

- Use stable identifiers in repeated UI rows instead of index-based iteration where practical.
- Clean up temporary local UI feedback state when it is no longer relevant.
- Replace deprecated AppKit drawing APIs such as `NSImage.lockFocus` only if the replacement is low-risk and preserves current logos exactly.
- Keep file splitting focused on readability; do not introduce abstractions that make SwiftUI flow harder to follow.

## Non-Goals

- No new providers.
- No custom/dynamic provider system.
- No provider data model or credential storage redesign.
- No major visual redesign.
- No heavy animation pass.
- No dashboard-style Console.
- No default folding of secondary providers unless review proves the extra interaction is worth it.

## Suggested Implementation Order

1. Baseline verification and visual inventory.
2. Microinteractions for refresh/error/status changes.
3. UI semantic fixes and accessibility labels.
4. Tone color cleanup.
5. `MenuBarContentView` split with behavior unchanged.
6. Console Home density and hierarchy tweaks.
7. Console API focus/menu copy polish.
8. Low-risk UI code hygiene.
9. Manual review, screenshot check, and release preparation.

## Acceptance Criteria

- Existing provider workflows still work: DeepSeek, Zhipu GLM Coding Plan, and Codex/ChatGPT.
- Existing menu bar, Console Home, Console API, and Settings layouts remain recognizable.
- UI feels smoother and clearer, not more decorative.
- No state is communicated by color alone.
- `swift run APIInquiryCoreTestsRunner` passes.
- `swift build` passes.
- Manual review covers:
  - menu bar panel
  - Console Home
  - Console API
  - Console Settings
  - Chinese and English text where changed
  - dark mode and any light-mode-sensitive colors

## Open Review Points

- Whether quota rows should stay vertical or get a compact horizontal variant.
- Whether secondary providers should remain always visible in the menu panel.
- Whether API deletion/removal confirmation should stay inline or move to a popover/sheet.
- Whether any leading status indicator on Home cards improves scanning enough to justify the extra visual element.
