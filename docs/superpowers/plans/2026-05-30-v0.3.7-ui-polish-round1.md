# v0.3.7 UI Polish Round 1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement the first low-risk v0.3.7 UI polish pass without changing existing provider workflows, layout identity, or v0.4.0 provider scope.

**Architecture:** Keep semantic state in `APIInquiryCore` only where the App target currently depends on localized display strings. Keep visual polish in the App target through existing SwiftUI views and `ConsoleDesignTokens`. Avoid new dependencies, broad view splits, dashboard-style layout changes, or provider model changes.

**Tech Stack:** Swift 5.9, SwiftUI/AppKit, macOS 13+, Swift Package Manager, `APIInquiryCoreTestsRunner`.

---

## Scope Boundary

Round 1 includes only refresh/error/status microinteractions, `APIAccessBadge` semantic state repair, concise help/accessibility labels, and lightweight tone color cleanup in the existing token layer.

Round 1 excludes `MenuBarContentView` splitting, Console Home redesign, API removal confirmation interaction changes, new providers, dynamic provider IDs, generic v0.4.0 provider capabilities, and any visual treatment that competes with quota/provider/status/credential content.
