# API Inquiry v0.3.7 UI 优化计划

> **给 agentic workers：** 按小步、可审查的方式实施本计划。除非某一步明确要求，否则必须保持当前产品风格和可见工作流不变。

**目标：** 在 v0.4.0 多供应商扩展之前，继续打磨应用 UI，让 API Inquiry 保持极简、克制、高效、轻科技。

**设计原则：** UI 必须为内容服务。任何动效、装饰、布局变化或视觉处理，只要会让状态、额度、供应商或凭据内容更难扫读，就应删除或拒绝。

**架构边界：** 保持 v0.3.6-Refactor 后的 Provider 架构不变。UI 修改应主要停留在 App target；Core 只在需要移除脆弱字符串判断或补充展示语义时做小范围 display state 调整。

**技术基线：** Swift 5.9、macOS 13+、SwiftUI/AppKit、本地 `APIInquiryCoreTestsRunner`。

---

## 风格守则

- 整体气质保持极简、安静、原生 macOS、轻科技。
- 菜单栏面板是快速查看状态的表面，不是完整状态看板。
- Console 是聚焦的管理表面，不是运维后台。
- 动效只用于状态反馈：加载、刷新完成、错误出现、小幅数值变化。
- 避免装饰性动效、强发光、大面积渐变、装饰阴影，避免任何与内容抢注意力的元素。
- 状态色只做语义表达，克制使用。
- 供应商品牌视觉保持可识别，但必须紧凑。
- 不引入新的第三方 UI 或架构依赖。

## 范围

### 1. 微交互与状态反馈

- 为刷新控件增加克制的 loading 动画。
- 为错误信息和恢复动作出现增加轻微过渡。
- 在当前已经使用状态色的位置，为 tone 变化增加自然过渡。
- `.contentTransition(.numericText())` 这类 macOS 14+ 能力必须做 availability 分支；macOS 13 保持安全兜底。

### 2. UI 语义与稳健性

- 修复 API access badge 依赖本地化字符串比较判断状态的问题，改成明确的展示状态。
- 改善菜单和图标按钮的 help/accessibility 文案，尤其是“更多操作”控件。
- 避免 UI 判断依赖本地化文案。

### 3. 颜色与可访问性打磨

- 在现有 token 层中集中 light/dark 安全的 tone color。
- 改善 warning 色可读性，但不让色彩变吵。
- 为 provider row、quota hero row、状态 badge、图标按钮补充精简 accessibility label/value。
- 颜色状态必须同时有文字语义，不能只靠颜色表达。

### 4. 菜单栏维护性

- 先拆分 `MenuBarContentView`，不改变视觉行为。
- 建议组件：
  - `MenuBarHeaderView`
  - `MenuBarQuotaHeroView`
  - `MenuBarStatusView`
  - `MenuBarSecondaryProvidersView`
  - `MenuBarFooterView`
- 除非审查中证明存在具体布局问题，否则暂时保留 320 pt 面板宽度。

### 5. Console Home 轻量层级优化

- 只在服务内容扫读时增强视觉层级。
- 如果 metric 信息密度过高，可考虑把低优先级更新时间从主指标行移出。
- 如果能明显提升扫读效率，可考虑非常克制的左侧状态指示。
- 不让 provider 卡片变成 dashboard 风格。

### 6. Console API 交互打磨

- 保持当前单行供应商管理结构。
- 为 API Key 编辑态增加焦点管理。
- 将更多菜单的 help/accessibility 文案从“移除供应商”改成“更多操作”。
- 删除/移除确认是否改成 popover/sheet 需要后置审查，因为这会改变交互模型。

### 7. UI 代码卫生

- 在可行位置用稳定标识替代基于 index 的重复 UI 渲染。
- 清理不再相关的临时本地 UI feedback 状态。
- 只有在低风险且能保持当前 Logo 效果完全一致时，才替换 `NSImage.lockFocus` 这类已废弃 AppKit 绘制 API。
- 文件拆分只服务可读性，不引入让 SwiftUI 流程更难理解的抽象。

## 非目标

- 不新增供应商。
- 不做自定义/动态 Provider 系统。
- 不重做 Provider 数据模型或凭据存储。
- 不做大规模视觉重设计。
- 不做重动效。
- 不把 Console 做成 dashboard。
- 不默认折叠 secondary providers，除非审查证明额外交互值得引入。

## 建议实施顺序

1. 基线验证和视觉现状清点。
2. 刷新、错误、状态变化的微交互。
3. UI 语义修复和可访问性标签。
4. Tone color 清理。
5. 拆分 `MenuBarContentView`，保持行为不变。
6. Console Home 密度和层级微调。
7. Console API 焦点与菜单文案打磨。
8. 低风险 UI 代码卫生。
9. 手动审查、截图检查和发布准备。

## 验收标准

- 现有供应商工作流仍正常：DeepSeek、智谱 GLM Coding Plan、Codex/ChatGPT。
- 菜单栏、Console Home、Console API、Settings 的现有布局仍然可识别。
- UI 变得更顺、更清楚，而不是更花。
- 没有任何状态只靠颜色表达。
- `swift run APIInquiryCoreTestsRunner` 通过。
- `swift build` 通过。
- 手动审查覆盖：
  - 菜单栏面板
  - Console Home
  - Console API
  - Console Settings
  - 变更过的中英文文案
  - 深色模式，以及任何受浅色模式影响的颜色

## 待审查点

- quota rows 是否保持纵向，还是引入紧凑横向变体。
- 菜单栏面板里的 secondary providers 是否继续默认展开。
- API 删除/移除确认是否保持内联，还是改为 popover/sheet。
- Home 卡片左侧状态指示是否真的提升扫读效率，是否值得增加这个视觉元素。
