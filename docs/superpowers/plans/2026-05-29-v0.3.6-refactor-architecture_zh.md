# v0.3.6 架构重构实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**目标：** 在保持当前功能和 UI 不变的前提下，将 API Inquiry 重构为更清晰的内置多 Provider 架构。

**架构：** 在 Core 中集中管理内置 provider 业务元数据，在 App target 中管理 provider 视觉元数据，让生产 ViewModel 统一依赖 coordinator，拆分 Console SwiftUI 巨型文件，并用现有 executable test runner 保护每一步行为。

**技术栈：** Swift 5.9 package、macOS SwiftUI/AppKit、Combine、本地 executable test harness `APIInquiryCoreTestsRunner`。

---

## 文件结构

- 创建：`Sources/APIInquiryCore/Providers/ProviderDescriptor.swift`，当 descriptor 扩展后避免 `ProviderCatalog.swift` 过宽。
- 创建：`Sources/APIInquiryCore/Providers/ProviderRegistration.swift`，承载 `descriptor + makeProvider` 内置注册。
- 创建：`Sources/APIInquiryCore/Providers/BuiltInProviderRegistry.swift`，让 catalog 顺序、默认 provider 和 app provider 构造共享同一来源。
- 修改 `Sources/APIInquiryCore/Providers/ProviderCatalog.swift`，扩展 descriptor 和元数据枚举。
- 修改 `Sources/APIInquiryCore/Providers/BalanceProvider.swift`，让 provider 实现不再持有展示元数据。
- 修改 `Sources/APIInquiryCore/Providers/DeepSeekBalanceProvider.swift`、`ZhipuCodingPlanProvider.swift`、`CodexQuotaProvider.swift`，只保留 fetch/parse/error 行为。
- 修改 `Sources/APIInquiryCore/Refresh/MultiProviderBalanceCoordinator.swift`，支持 descriptor-backed runtime 和并发刷新。
- 修改 `Sources/APIInquiryCore/ViewModels/MenuBarBalanceViewModel.swift` 与 `UsageConsoleViewModel.swift`，删除生产 single-provider 路径。
- 拆分 `Sources/APIInquiryCore/Formatting/ProviderDisplayFormatter.swift` 为更聚焦的 formatter/tone 文件。
- 在 `Sources/APIInquiryApp/` 创建 App 层 visual/style 文件。
- 将 `Sources/APIInquiryApp/UsageConsoleView.swift` 拆成聚焦的 Console 组件。
- 更新 `Sources/APIInquiryCoreTestsRunner/` 中的测试。

- [ ] **Step 1: 编写 descriptor 和 registry 完整性测试**

增加断言：每个 descriptor 的 `id` 唯一、`credentialAccount` 唯一、display/menu 文案非空、homepage URL 有效，并且具备 detail kind、credential management policy、access purpose、menu title prefix policy 和 secondary display name policy。显式验证 DeepSeek 是 prepaid balance，Zhipu 是 app-managed plan access，Codex 是 local-config quota access。断言 `ProviderID.allCases == ProviderCatalog.default.descriptors.map(\\.id)`，并且 built-in provider registration 顺序等于 catalog 顺序。

- [ ] **Step 2: 运行 RED**

运行：

```bash
swift run APIInquiryCoreTestsRunner
```

预期：因为 `ProviderCredentialManagement` 和 `ProviderAccessPurpose` 尚不存在而失败。

- [ ] **Step 3: 增加 metadata enum 和 descriptor 字段**

实现：

```swift
public enum ProviderCredentialManagement: Equatable {
    case appManagedAPIKey
    case localExternalConfiguration

    public var supportsConsoleCredentialManagement: Bool {
        self == .appManagedAPIKey
    }
}

public enum ProviderAccessPurpose: Equatable {
    case prepaidBalance
    case planQuota
}
```

将 `credentialManagement`、`accessPurpose`、`menuTitlePrefix` 和 `secondaryDisplayName` 加入 `ProviderDescriptor`。

- [ ] **Step 4: 增加内置 provider registration**

增加 registration 类型，让 factory closure 不进入可比较的 descriptor：

```swift
public struct ProviderRegistration {
    public let descriptor: ProviderDescriptor
    public let makeProvider: () -> BalanceProvider
}
```

增加 `BuiltInProviderRegistry.default`，按现有顺序注册 DeepSeek、Zhipu 和 Codex。`ProviderCatalog.default` 应从 registry 派生 descriptors。

- [ ] **Step 5: 将展示元数据从 provider 移出**

将 `BalanceProvider` 改成只要求：

```swift
public protocol BalanceProvider {
    var id: ProviderID { get }
    func fetchSnapshot(apiKey: String) async throws -> ProviderSnapshot
}
```

从具体 provider 类中删除 `displayName`、`menuPrefix`、`credentialAccount`、`homepageURL` 和 `supportsConsoleCredentialManagement`，保留 `id` 和请求/解析行为。
