# API Inquiry v0.3.2 Localization and Language Switching Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add full Chinese UI localization and `Auto / 中文 / English` language switching to API Inquiry while keeping the existing DeepSeek, Zhipu, and OpenAI/Codex provider logic unchanged.

**Architecture:** Add a lightweight localization layer to `APIInquiryCore` for language preference, system language resolution, and UI copy. Core ViewModels/Formatters should emit localized strings, while SwiftUI remains mostly responsible for layout and simple controls. Persist the language preference in `UserDefaults`, default to `Auto`, and use shared observable state so menu details and Console update after language changes.

**Tech Stack:** Swift 5.9, SwiftUI, AppKit, Combine, UserDefaults, Swift Package Manager, local `APIInquiryCoreTestsRunner`.

---

## Scope

### Included

- Add `AppLanguage.auto / zh / en`.
- Add a persisted language preference store with `Auto` as the default.
- Resolve `Auto` from macOS preferred languages: Chinese preferred language -> Chinese UI; otherwise English UI.
- Add a unified copy layer for menu details, Console Home, Console API, buttons, statuses, errors, settings feedback, last-updated text, and reset text.
- Add a language setting entry in Console, preferably near the top of Home or in a lightweight Settings area.
- Apply manual language changes to menu details and Console without restart where practical.
- Update README, release notes, and `Scripts/version.env` for v0.3.2.

### Excluded

- No additional languages.
- No translation of provider brand names.
- No changes to provider fetching, credentials, quota/balance parsing, or history data.
- No historical charts or CSV export.
