# API Inquiry v0.3.6-Refactor

## 应用优化

- 在 UI 验证后刷新本次重构发布，补入后续 provider runtime 收口。
- 将内置 Provider 元数据集中到 descriptor 和 registration，供应商显示名、菜单栏前缀、凭据账户、接入用途、凭据策略和主页现在都从同一来源派生。
- 让 `ProviderRegistration` 成为运行时 descriptor 来源：`MultiProviderBalanceCoordinator` 现在从 registrations 构建 runtime，不再通过全局 catalog 或 provider metadata fallback。
- 强化 provider runtime 默认值：自定义 registration 集合可以省略 `defaultProviderID`，coordinator 会从第一个 registration 推导；显式传入未注册 default 时，会在执行 provider factory 前拒绝。
- 让 `BalanceProvider` 专注于 API 请求，移除 protocol extension 中的 descriptor、显示信息和凭据 metadata convenience。
- 明确 provider factory 约定：应保持轻量，并避免凭据、网络或持久化存储副作用。
- 将 descriptor 拥有的 credential account 显式注入 `BalanceRefreshController`，刷新逻辑不再向 provider 查询 Keychain account。
- 将菜单栏和 Console ViewModel 统一到 `MultiProviderBalanceCoordinator`，从生产代码中移除历史 single-provider 路由路径。
- 拆分过大的 Console UI 文件，将导航、section header、Home 供应商行、API 供应商面板、Settings、badge、logo、指标格和设计 token 拆成独立 SwiftUI View。
- 将 provider 显示格式拆分为数值格式、状态文案、tone 判断和显示模型，同时保留 `ProviderDisplayFormatter` 作为兼容门面。
- 移除基于 provider 的 formatter overload，并简化 provider mock，避免测试继续暗示 metadata 存在于 provider 实例上。
- 将供应商图片/logo 逻辑从 DeepSeek 专属 helper 收敛到 `ProviderVisualCatalog`。
- Settings 页面版本号显示为本次重构构建标签 `v0.3.6-Refactor`。

## 行为改进

- 多 Provider 刷新现在并发执行，同时每个 Provider 继续保持独立的 loading/error 状态隔离。
- Codex 配置目标查询默认变为只读；需要创建目录的路径改为显式命名 API。
- 当 provider descriptor metadata 异常缺失时，Console 凭据管理动作现在默认拒绝，而不是乐观允许。
